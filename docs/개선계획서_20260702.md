## 3. 패키지 A — `/topics` 밸브 정비 + 리포트 성능 (통합 트랙, 최대 패키지)

> `/topics/[tag]`·`/reports/[procedure]`·`procedure-report.ts` 를 여러 항목이 함께 건드리므로 한 에이전트가 순차 처리(다른 패키지와는 병렬).

### ① 쉬운 말 설명
`/topics/{시술명}` 은 검색·AI 로 들어온 외부 방문자를 받는 밸브 페이지입니다. 지금은 (1) 제목 문장이 길고 투박하며, (2) 시술 후기로 가는 연결이 밋밋한 한 줄 텍스트 링크뿐입니다. 이를 (1) 제목 간결화, (2) `/reports` 에서 쓰는 **닫힌 리포트 글상자**(재시술% 히어로 + 통증·만족도 미니 + 헤드라인 한 줄)를 그대로 위에 얹는 방식으로 바꿔, 외부 방문자가 첫 화면에서 "데이터가 있는 곳"임을 즉시 느끼게 합니다. 겸해서 같은 파일들의 성능 결함(같은 계산 2번 실행, 캐시 없는 전체 점수계산)도 이 트랙에서 한 번에 고칩니다.

### ② 구체 수정 명세

**A-1. 주소 직접 진입 `/topics` → 홈 308 리다이렉트** (원장님 안 채택)
- `next.config.ts` redirects 에 `{ source: "/topics", destination: "/", permanent: true }` 1줄 추가. (기존 `/search → /` 와 동일 패턴, 엣지에서 처리라 렌더 비용 0)
- 참고: 현 404 자체는 SEO 불이익이 크지 않지만(색인 안 될 뿐), 내부 링크가 그리로 향하던 것이 문제였습니다. 리다이렉트가 안전망이 됩니다.

**A-2. `/reports` 사이드바의 generic `/topics` 링크 제거**
- `ReportsIndexSidebar.tsx:138-146` 의 "이 시술이 궁금하면 → 전문의 Q&A" 링크 블록 삭제.
- 근거: PRD §4.3 의 "얇은 링크"는 **시술 단위**(topics/{tag} ↔ reports/{ko}) 설계입니다. 인덱스 사이드바의 시술 맥락 없는 generic 링크는 설계 밖이었고, 밸브 원칙(내부에서 /topics 진입 없음)과도 맞지 않습니다. 시술 단위 얇은 링크는 그대로 유지됩니다.

**A-3. `/topics/[tag]` 제목 간결화** (원장님 질문 "없애도 되지 않을까?" 에 대한 답)
- 현재: `TopicTagView.tsx:56-58` h1 = "피부과 전문의가 답한 #써마지 관련 글 90개".
- **권고: 문장은 없애되 h1 자체는 짧게 유지** — `#써마지 전문의 Q&A` 로 축약(글 수 카운트 문구 삭제).
- 이유(근거): h1 은 검색엔진·AI 가 "이 페이지의 주제"를 읽는 대표 신호라 **완전 삭제는 밸브 페이지의 SEO 목적과 상충**합니다. 반면 "관련 글 N개" 같은 부가 문장은 신호 가치가 없어 없애도 무방합니다. `<title>` 메타("써마지 Q&A 총정리")는 별개라 그대로 유지됩니다.
- 완전 삭제를 원하시면 그렇게도 가능합니다(결정 D-2) — 그 경우에도 색인은 유지되나 주제 신호가 한 단계 약해집니다.

**A-4. 얇은 링크 → 닫힌 리포트 글상자 임베드** (원장님 안 채택)
- 현재: `TopicTagView.tsx:63-79` "이 시술 후기 N건 보기 →" 텍스트 링크.
- 변경: `/reports` 인덱스의 접힘 카드(ReportsIndexCard ①요약부 — 카테고리 틴트 헤더 + 시술명 + 재시술% 히어로 + 통증·만족도 미니 + 회전 헤드라인)를 **공용 컴포넌트로 추출**(`ReportSummaryBox`)해 그 자리에 배치. 클릭 시 인라인 펼침이 아니라 **`/reports/{시술}` 로 이동**(밸브 → 리포트 유입 동선 유지).
- 데이터: 추가 DB 비용 **0** — topics 가 이미 호출하는 경량 풀 RPC(`get_review_summary_pool`)에 필요한 값(만족도·통증·재시술 3분류·카테고리·후기수)이 전부 있습니다. `getReportSummaryForTag` 반환을 `{count}` → 풀 row 요약 전체로 확장하고, 헤드라인은 `/reports` 와 동일한 `report-headline` 로직을 서버에서 재사용(SSR/CSR 일치). 카테고리는 한글→slug 매핑(기존 getProcedureReport 매핑 재사용).
- ⚠️ **PRD 결정 변경 동반**: PRD §4.3 은 "자기잠식 방지로 /topics 에 리포트 카드 미노출, 얇은 링크만"(2026-06-05) 이라고 명시합니다. 이번 변경은 그 결정의 갱신입니다 — 닫힌 상자는 여전히 /reports 로 보내는 입구(전문 열람은 리포트에서만)라 자기잠식 취지는 유지된다고 판단되나, **결정 주체가 원장님이므로 D-3 승인 시 PRD 문구도 함께 갱신**합니다.

**A-5. 성능 — topics 캐시** (감사 H2)
- `topics/[tag]/page.tsx` 의 `fetchFeedSidebarData`(19-40행)가 캐시 없이 매 요청 `feed_cards_scored` 300행 전체 점수계산 실행 → 홈이 이미 쓰는 `unstable_cache`(5분) 공용 모듈로 추출(`lib/popular-tags-cached.ts` 신설, 홈 `page.tsx:39` 도 import 로 교체)해 공유.
- generateMetadata(112행)와 본문 fetchPostsForTag(93행)의 **동일 count 쿼리 중복**도 React `cache()` 로 1회화.

**A-6. 성능 — 리포트 상세 중복 제거** (감사 H3)
- `reports/[procedure]/page.tsx`: `resolveProcedure`·`getProcedureReport` 가 메타 생성과 본문에서 각 1번씩(총 2번) 실행 → **ko 만 인자로 받는 React `cache()` 래퍼**로 감싸 1회화. ⚠️ supabase client 객체를 인자로 받으면 캐시가 안 걸리므로 client 는 래퍼 내부 생성(기존 `reports-pool.ts:13` 패턴).
- `procedure-report.ts`: family RPC 를 `cache()` 로 공유해 `getProcedureReport` ↔ `getFamilyReviewCardIds` 중복(2회→1회) 제거, 내부 독립 3쿼리(분류·앵커·인구통계) `Promise.all` 병렬화.
- 효과: 요청당 DB 왕복 약 **12회 → 약 5회**.

**A-7. 리포트 상세 SEO 게이트** (결정 D-1)
- 후기 4건 미만 시술 상세는 `robots:{index:false, follow:true}` + JSON-LD `aggregateRating` 생략(후기 1건짜리 별점이 구글에 노출되는 리스크 제거). 임계는 기존 `FEED_MIN_REVIEWS`(=4) export 재사용 — 허브와 한 기준.
- 같은 파일 내 소소한 정리: JSON-LD `datePublished`(앵커 created_at)·`dateModified`(최신 후기 시각) 채움, 낡은 주석 정정.

### ③ 무엇이 좋아지는가
- 외부 유입 방문자의 첫 화면: 긴 문장 대신 **깔끔한 제목 + 데이터가 살아있는 리포트 상자** → 리포트로의 전환 동선 강화.
- `/topics` 직접 진입 404 소멸(홈으로 안착), 리포트 허브의 죽은 링크 제거.
- 리포트 상세·태그 페이지 속도 개선 + DB 부하 절반 이하, 봇 크롤 시 반복 계산 차단.
- 얇은 별점(후기 1~3건)의 구글 노출 리스크 제거.

### ④ 검증
빌드 + `/topics/써마지` 렌더(상자 표시·클릭 이동) + `/topics` 308 확인 + schema-auditor(h1 변경·게이트·datePublished) + 응답시간 전후 비교.

### ⑤ 리스크
중간. 화면 개편(A-3·A-4)은 시각 확인 필요 — 완료 후 스크린샷 보고. A-4 는 PRD 결정 변경이라 D-3 승인 필수.

## 5. 패키지 C — 보안 하드닝 (자연어 설명 보강판)

> 공통 전제: 네 건 모두 "지금 뚫려서 피해 중"이 아니라, **이론상 열려 있는 틈을 미리 잠그는** 작업입니다. 사용자 화면·관리자 화면은 아무것도 바뀌지 않습니다.

**C-1. 관리자용 유튜브 연동 화면의 문자열 그대로 출력 (XSS)**
- 쉬운 말: 그 화면은 주소 뒤에 붙는 문자열(`?error=...`)을 **화면에 그대로** 출력합니다. 누군가 그 자리에 악성 스크립트를 담은 링크를 만들어 관리자에게 보내고 관리자가 클릭하면, 그 스크립트가 **관리자 브라우저 안에서 실행**되어 관리자 세션(로그인 권한)을 훔칠 수 있습니다.
- 수정: 특수문자를 "글자 그대로"만 보이게 이스케이프 처리. 화면 변화 없음.

**C-2. 정기작업(cron) 비밀키 비교 방식**
- 쉬운 말: 서버의 자동 작업(검색엔진 알림, 알림 발송 등)은 비밀키가 맞아야 실행됩니다. 지금 비교 방식은 "첫 글자부터 틀리면 즉시 거절, 많이 맞을수록 늦게 거절"이라, 공격자가 **응답 속도의 미세한 차이를 재면 비밀키를 한 글자씩 알아낼 이론적 여지**가 있습니다.
- 수정: 몇 글자가 맞든 항상 같은 시간이 걸리는 표준 비교 방식으로 교체(이미 다른 두 곳은 이 방식 사용 중 — 통일하는 것).

**C-3. 브라우저 자동 보안보고 창구에 횟수 제한** *(= "글 신고"가 아닙니다)*
- 쉬운 말: 사용자가 누르는 "글 신고"가 아니라, **방문자의 브라우저가 자동으로 보내는 기술 보고서**(보안규칙 위반 감지 시 자동 전송) 접수 주소입니다. 로그인 없이 누구나 무한정 보낼 수 있어, 악의적으로 초당 수천 건을 쏘면 서버 기록이 쓰레기로 덮이고 무료 플랜의 함수 호출 한도가 소진될 수 있습니다.
- 수정: 같은 IP 는 분당 N건까지만 접수.
- 참고: 원장님이 말씀하신 **콘텐츠 신고(/api/reports)는 확인 결과 이미 횟수 제한이 걸려 있습니다** — 추가 조치 불필요.

**C-4. 병원 검색 입력 길이 제한**
- 쉬운 말: 병원 검색창 뒤의 네이버 API 호출에 글자수 제한이 없어, 수천 글자를 넣어 네이버 무료 사용량을 소진시킬 수 있습니다. 100자 제한 추가.

**수정 파일**: `lib/auth/timing.ts`(신규 공용 비교 헬퍼) · cron 3라우트 · `youtube-oauth/callback`(이스케이프 + state 비교 교체) · `csp-report` · `place-search`. **리스크**: 낮음(판정 결과 동일, 방식만 교체).

## 6. 패키지 D — DB 하드닝 (supabase-specialist)

**D-1. `get_research_panel` 함수 잠금 — 관리자는 그대로 다 봅니다 (질문 주신 부분)**
- 쉬운 말: 관리자 통계 **페이지**는 이미 잠겨 있습니다(비관리자는 못 들어감). 문제는 그 페이지가 내부적으로 부르는 **DB 함수 자체**가 별도의 신분 확인 없이 "로그인한 사람 누구나" 호출 가능하게 열려 있다는 점입니다. 비유하면 관리자실 문은 잠겼는데, 그 안 서류함으로 통하는 **뒷문**이 있어서 기술 지식이 있는 일반 회원이 주소만 알면 전체 회원 수·활성 사용자 수·후기 작성자 수를 직접 조회할 수 있습니다.
- 수정: 함수 첫 줄에 "호출자가 관리자인지 확인, 아니면 거절"을 추가. 같은 부류의 다른 통계 함수들은 이미 이 확인이 있는데 이 함수 하나만 빠져 있었습니다(감사에서 발견).
- **관리자 화면·통계는 단 하나도 바뀌지 않습니다.** 막히는 건 "관리자가 아닌 사람의 뒷문 호출"뿐입니다.

**D-2. 민감정보 컬럼 명시 잠금**
- 쉬운 말: 최근 추가된 민감 컬럼들(피부 광반응 타입, 각종 동의 시각·버전)은 지금도 비로그인 조회가 차단돼 있지만, "명시적으로 잠근 것"이 아니라 "기본값 덕분"입니다. 나중에 설정을 되돌리는 실수가 나오면 노출될 수 있어 명시적 잠금(REVOKE)을 겁니다. 현재 동작 변화 0.
- **D-3(조건부)**: follows 테이블·헬퍼 함수 잠금 2건은 기존 접근 경로를 전수 확인해 **안전할 때만** 포함(확인에 걸리면 제외하고 보고).
- 적용: 마이그레이션 `0325_security_hardening.sql` 1개, Management API UTF-8 경로, 적용 직후 차단 재검증.
