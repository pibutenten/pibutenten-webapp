## 구현 착수 전 정정 (2026-07-05 · 3중 독립검수 + 운영 DB 실측)

> 종합 판정 = **go_with_changes**. 설계 방향(명함 단위·비귀속 다대다·SECURITY DEFINER RPC·스냅샷 동의)과 사실관계는 대부분 정확하고 **근본 재설계는 불필요**하다. 단 **구현 즉시 실패하는 blocker 1건(role enum)** + **착수 전 계획 반영 high 9건**을 아래대로 반영한 뒤 착수한다. (독립검수 판정: 1인 blockers_exist, 2인 go_with_changes → 총괄이 운영 DB 실측으로 blocker 재분류.)

### C. clinic_id 설계 확정 (원장 지시 2026-07-05)
- **건보(심평원) `clinics`(요양기관 코드+이름) = 원본 참조 데이터. 절대 수정 금지.** → §1.1 "표시명(→ `clinics.name`)" **철회**(건보 이름 rewrite 안 함).
- **새 FK `clinic_id` 하나**를 정확히 만들어 건보 코드 참조: `doctors.clinic_id → clinics(id)`, 병원 명함 `profiles.clinic_id → clinics(id)`. 연결·시술노트 드롭다운·소속 판정은 **전부 이 필드로만**.
- 지점 표시명("힐하우스피부과의원 강남점") = **병원 계정 명함 `display_name`** (clinics 불변).
- 지점 ↔ 코드 매핑(16957=강남 등)은 건보 기준 원장 확정값(§1.1) 사용. **doctors backfill은 기존 `branch` 텍스트('강남점' 등) → clinic_id** 로 매핑(clinics 이름이 동일해 이름 매핑 불가).
- **지점↔코드 주소 교차확인 완료(2026-07-05):** 판교=16955(성남 분당 백현동)·건대=16956(서울 광진 능동로)·강남=16957(서울 강남 강남대로)·대구=16958(대구 동구)·수원=16959(수원 팔달). 건보 등록 의사수(2/2/4/3/4)도 원장 15명 분포와 일치. §1.1 매핑 정확.
- **실명(2026-07-05 지시):** `profiles.legal_name` **선택 복원**(회원이 온보딩에서 입력하면 유지 — 향후 쇼핑몰 주문·결제·배송 등에 사용). 회원이 안 넣어도 됨. **병원 매칭은 회원 legal_name에 의존하지 않음** — 실명은 **병원이 입력**해 `clinic_member_links.requested_legal_name/patient_name`에 저장, 회원은 동의 화면에서 본인 확인. **매칭 하드키 = handle + 생일**(회원 legal_name 있으면 소프트 대조 가능).

### D. 진짜 blocker (구현 즉시 실패 — 필수)
- **B1. role은 ENUM (§5.1).** `ALTER TYPE user_role ADD VALUE 'clinic'`는 ① 트랜잭션(BEGIN/COMMIT) 안에서 실행 불가, ② 같은 트랜잭션에서 새 값 사용 불가. **0342를 Management-API 2회 호출로 분리**: (1) `ALTER TYPE ... ADD VALUE IF NOT EXISTS 'clinic'` 단독 → (2) 별도 트랜잭션에서 나머지 DDL(clinic_id·legal_name 선택 복원·is_clinic·reserved_handles 등). `developer` 추가 선례 마이그 패턴 재사용.

## 0. 핵심 원칙

1. **명함 단위.** 귀속·권한·소유는 사람이 아니라 **명함(`profiles.id`) 단위**(ADR 0012). 병원 계정도 명함.
2. **비귀속 · 다대다.** 병원-환자 연결은 독점 아님. 한 회원이 여러 병원과 연결 가능.
3. **광고 아님 · 병원 비노출.** 유인·의료광고 아님. **치료 병원명 공개 화면 비노출.**
4. **명칭 = "시술노트"**(공식 텀). "차트/진료기록" 금지(의료법 §21). 병원이 회원의 시술노트를 대신 작성.
5. **동의 = 알림 → 온보딩형 전체화면.** 강제 팝업 아님. 기본 미동의(opt-in).
6. **동의 = 이중 동의.** ① 병원이 내 시술노트 대신 저장 + ② 내 정보(아이디·이름·생일·이메일·피부정보)를 병원에 **제공**(개인정보보호법 §17).
7. **정보는 스냅샷.** 동의 시 1회 복사 → 병원이 수정, 회원 원본과 동기화 안 됨, 병원만 조회.
8. **하드삭제 금지.** 원장 퇴사·연결 해제 등은 **비활성**으로 처리하고 과거 콘텐츠 전부 보존(ADR 0002/0012).
9. **전화번호는 앱에서 수집 안 함(현행 유지).** 앱에 있는 것만 제공. 전화·집주소는 병원이 직접 입력(선택).

## 2. 원장(doctor) 계정 모델 · 생애주기 (관리자 프로그램 개편 포함)

### 2.1. 세 개의 독립 토글 (`is_active` 폐기 — 각각 별도, 전부 불리언)
- **근무 병원 = `doctors.clinic_id`**(→clinics): 어느 지점 → **시술노트 드롭다운** 대상. 이동=변경(과거 기록 불변).
- **재직/소속 = `doctors.is_affiliated`**(bool, 기본 true): 퇴사 시 `false` → **드롭다운에서 제외**(퇴사자에겐 새 시술노트 못 붙임).
- **공개 = `doctors.is_listed`**(bool, 기본 true): 공개 페이지(`/doctors/[slug]`) on/off. **퇴사와 독립** — 관리자가 따로 결정.

| 상태 | clinic_id | is_affiliated | is_listed | 드롭다운 | 공개 페이지 |
|---|---|---|---|---|---|
| 재직+공개 | 지점 | true | true | 뜸 | 뜸 |
| 재직+비공개(현 미활동 6명) | 지점 | true | false | 뜸 | 안 뜸 |
| 퇴사+공개 유지 | 지점 | false | true | 안 뜸 | **뜸** |
| 퇴사+비공개 | 지점 | false | false | 안 뜸 | 안 뜸 |

- **드롭다운** = `clinic_id=계정 AND is_affiliated`. **공개 페이지** = `is_listed`(퇴사 무관, 독립).

### 2.2. 일관된 등록 방식 (계정 없는 경로 폐기)
- **모든 원장 = 개인 소셜 가입 → 관리자가 원장 명함 연결**(기존 `admin_create_doctor_profile`, ADR 0012 묶음). 활동/비활동은 `is_listed` 로만 조정.
- `doctors` = 원장 정체 SSOT. `profiles.doctor_id` = 계정 연결. `clinic_id` = 소속. `is_listed` = 공개.

### 2.3. 생애주기
- **등록:** 개인 가입(온보딩 완료) → 관리자 "원장 명함 연결" → `doctors` 행 + 원장 명함 생성, `clinic_id` 세팅, `is_listed`(활동이면 on).
- **이동(다른 지점 오픈 등 — 피부텐텐 활동 유지):** `clinic_id` 만 변경. 이전 지점 드롭다운에서 빠지고 새 지점에 뜸. `is_affiliated`·`is_listed` 는 그대로. **과거 시술노트 담당 원장 불변**(`diaries.doctor_id`, 소급 수정 없음).
- **퇴사·탈퇴(완전 퇴사 or 비피부텐텐 병원 이직 — 하드삭제 금지):** `is_affiliated=false` → **드롭다운에서만 제외**(`clinic_id` 는 이력으로 둠). **공개 페이지는 별도 결정** — `is_listed` 유지 시 과거 Q&A 페이지 남김, off 시 내림(관리자 선택). 원장 명함도 비활성. **개인 계정·과거 시술노트·Q&A·댓글 전부 보존.**

### 2.4. 관리자 프로그램 개편 (정합성 · SSOT — 이번 범위, 조사 확인)
현재: 등록=`/admin/users/[id]` 회원전환(super admin), 편집=`/admin/doctors/[slug]/edit`(super admin **또는 원장 본인**)에서 `profile_data`(경력·SNS 등)만. **부족: clinic_id·is_listed·퇴사 없음, name/slug 수정 없음, 삭제 없음.**
개편:
- **관리자 전용 설정**(super admin only, 원장 본인 수정 금지): `clinic_id`(근무지점/이동)·`is_affiliated`(재직/퇴사)·`is_listed`(공개) — 3개 독립 토글. 신규 `PUT /api/admin/doctors/[slug]/settings` + `requireAdmin`(super). 원장 연결 폼에도 `clinic_id`·`is_listed` 추가. **`admin_create_doctor_profile`(0192)에 `p_clinic_id`·`p_is_listed` 파라미터 추가 필요**(0341 CREATE OR REPLACE, 정정 §E-H8).
- **이름(name) 수정 + SSOT 동기화**: `doctors.name`(신원 SSOT)과 원장 명함 `profiles.display_name` 이 현재 따로 놀 수 있음(불일치 발견). super admin 이 name 수정 시 원장 명함 `display_name` 동시 갱신.
- **slug 편집(2026-07-05 지시)**: 미활동 6명 등록 시 원장 슬러그를 운영 프로그램에서 제대로 받아 설정·정정할 수 있어야 함. → **`slug` 는 미공개(`is_listed=false`) 동안 super admin 편집 가능**(신규 slug-edit 경로), **공개(`is_listed=true`) 후 잠금**(URL·SEO 안정). 신규 등록 slug 는 `admin_create_doctor_profile`(p_slug)로 이미 설정.
- **원장 본인:** `profile_data`(경력·SNS) 자기 편집 유지. 소속·공개·퇴사는 못 바꿈.
- **삭제 없음**(하드삭제 금지) — 퇴사=`is_affiliated=false`. 관리자 목록(`/admin/doctors`)에 근무지점·재직·공개 상태 표시 + super admin 토글.

## 3. 사용자 시나리오

### 3.1. 회원
1. 방문 시 카운터에 **자기 아이디(handle)** 알려줌.
2. 병원이 등록 요청 → **알림** → 눌러 **온보딩형 동의**(제공 항목 고지) → **동의** → 연결 성립 + 내 정보 1회 제공(스냅샷).
3. 이후 방문 시 병원이 시술노트 작성 → **자동으로 내 기록 추가** + 알림.
4. 내 기록(`/notes`) 확인 → **후기 작성**(`/review/new`, 시술 prefill).
5. 통제: 개별 기록 삭제 / 병원 연결 해제.

### 3.2. 병원(지점 계정)
1. 구글 로그인 → **병원 대시보드** `/clinic`.
2. **환자 등록(최초):** 아이디+이름+생년월일 → 등록 요청. (등록번호 등 병원 자체 항목도)
3. **동의되면:** 회원 정보 스냅샷 수신 → 환자 기록 생성(병원 수정 가능).
4. **시술노트 작성(이후 매번):** 환자 골라 시술노트 폼(회원 폼과 동일) 입력 → 저장. 원장 칸=그 지점 원장 드롭다운.

## 4. 매칭 & 동의

### 4.1. 매칭 = 아이디 + 생일(하드 대조) + 실명(병원 입력·회원 확인)
- **1차 키 = handle**(고유). **2차 하드 대조 = 생일**: 서버가 회원 저장값과 대조, 불일치 거부(값 미반환). **실명은 병원이 입력**(회원 온보딩 미입력 가능 — 정정 §C) → `clinic_member_links.requested_legal_name/patient_name`에 저장하고 **회원이 동의 화면에서 본인 확인**(서버가 회원 저장 실명과 강제 대조하지 않음). 병원에 회원 원본 미표시.

### 4.2. 동의 = 온보딩형 · 이중 동의 · 병원별 1회 · 이후 자동
1. **등록 요청:** 병원 [아이디+이름+생일(+등록번호)] → 대조 → `pending` + `clinic_link_request` 알림.
2. **동의:** 회원 알림 클릭 → `/onboarding/clinic-link/[id]` 전체화면 동의 → `active` + `consent_at` + **회원 정보 스냅샷 복사**. 거절=`rejected`.
3. **시술노트 작성(이후):** `active` 상태에서 병원이 환자 골라 저장(`source='clinic'`) + `clinic_visit_added` 알림. 재동의 없음.
4. 회원은 병원별 연결 해제(`revoked`)·개별 기록 삭제.

### 5.1. `profiles` (0342)
```
role += 'clinic'   -- ⚠ user_role 은 ENUM(현 admin/doctor/user/developer). ALTER TYPE ADD VALUE 는
                   --   트랜잭션 밖 단독 + 새 값은 별 트랜잭션에서 사용(정정 §D-B1). 0342 를 2-call 로 분리.
clinic_id  bigint NULL → clinics(id)     -- 건보 코드 참조(불변). role='clinic' 명함 소속 지점(doctor_id 와 대칭)
```

### 5.2. `doctors` 확장 (0341)
```
clinic_id     bigint  NULL → clinics(id)    -- 근무 지점(건보 코드 참조, 불변). 9명 backfill=기존 branch 텍스트('강남점' 등)→clinic_id 매핑(정정 §C). 이동=변경(과거 기록 불변)
is_affiliated boolean NOT NULL DEFAULT true  -- 재직/소속. 퇴사 시 false → 드롭다운 제외
is_listed     boolean NOT NULL DEFAULT true  -- 공개 페이지 on/off (퇴사와 독립)
```
- 드롭다운 = `doctors WHERE clinic_id=계정 AND is_affiliated`. 공개 페이지 = `is_listed`(퇴사와 독립).
- clinic_id·is_affiliated·is_listed·name = **super admin 전용**(원장 본인 수정 금지, §2.4). name 수정 시 원장 명함 `display_name` 동기화, `slug` 잠금.
- **공개 필터(is_listed) 추가 지점(운영 DB·grep 재확인 — 정정 §E-H2):** `/doctors` 목록 · `/doctors/[slug]` · `/doctors/[slug]/[year]/[postSlug]` · 홈 원장수(`page.tsx`) · `sitemap.ts` · **`about/page.tsx`(JSON-LD member)** · **`rss/route.ts`** · **`api/cron/indexnow/route.ts`**. 관리자 목록(`admin/doctors`·`admin-card-extras.ts`)은 전체 표시 + 비공개 배지. **확정: 미공개(is_listed=false)=없는 페이지** — `/doctors/[slug]`·`[year]/[postSlug]` `notFound()`(404) + `middleware.ts` 존재검사(321행)·`[handle]`·`post-data.tsx` 일관 404. 검색/피드 RPC는 후순위.

### 5.3. `diaries` 확장 (0343) — ★아래 5컬럼 전부 신규(운영 DB 부재 확인). 기존 diaries 컬럼은 보존.

### 5.4. `clinic_member_links` (0344) — 연결 + 병원 환자 기록(명함 단위)
```
id bigserial PK
clinic_id  bigint NOT NULL → clinics(id) ON DELETE CASCADE
profile_id uuid   NOT NULL → profiles(id) ON DELETE CASCADE   -- ★ 회원 명함
status enum('pending','active','rejected','revoked')
created_by_clinic_profile_id uuid NOT NULL → profiles(id)     -- 등록한 병원 계정
consent_at timestamptz NULL · consent_version text NULL
requested_legal_name text NULL · requested_birthdate date NULL   -- 대조용(감사, 반환 금지)
registration_number text NULL          -- 병원 등록번호(그 병원 내부, 고유값 아님)
patient_phone text NULL · patient_address text NULL             -- 병원 직접입력(앱에 없는 값)
patient_name text NULL · patient_birthdate date NULL · patient_email text NULL   -- 회원→병원 제공 스냅샷(병원 수정 가능)
patient_skin_profile jsonb NULL        -- {gender,skin_type,skin_concerns,face_shape,fitzpatrick,interested_procedures} 스냅샷
created_at timestamptz · revoked_at timestamptz NULL
UNIQUE(clinic_id, profile_id) WHERE status='active'   -- 다대다
```
- **테이블 직접 GRANT 없음 → RPC 전용.** 병원 환자 기록은 그 병원만 조회. 스냅샷=동의 시 profiles 에서 복사(라이브 아님).

## 6. RPC (0345) — 모두 SECURITY DEFINER, `/api/visits` 패턴 계승

1. **`clinic_request_link(p_handle, p_legal_name, p_birthdate, p_registration_number?)`** — 병원: `is_clinic`+rate-limit → 대조 → `pending` + 알림.
2. **`member_respond_link(p_link_id, p_consent, p_backfill_profile)`** — 회원: active/rejected + consent + **스냅샷 복사**.
3. **`clinic_add_visit(p_link_id, p_procedures, p_visited_on, p_doctor_id?, p_doctor_name?, p_manager_name?, p_note?, p_diary_body?, p_next_appointment_date?)`** — 병원: `active` 필수 → `diaries`(source='clinic', clinic_id, doctor_id) + 알림. `p_doctor_id` 는 `doctors.clinic_id`=계정 검증.
4. **`clinic_update_patient(p_link_id, …)`** — 병원: 환자 기록(등록번호·전화·주소·스냅샷) 수정.
5. **`get_clinic_patients(p_search?)` / `get_clinic_patient(p_link_id)`** — 병원: 자기 환자 목록·상세.
6. **`member_revoke_clinic_link(p_link_id)`** — 회원.
7. **`member_delete_clinic_visit(p_visit_id)`** — 회원(`source='clinic'` 한정).

> 기존 `create_visit_with_entries`(0297)는 본인 명함 전제라 재사용 안 함. `create_diary` 패턴 계승.

## 7. RLS · 보안
- **격리:** 병원은 `source='clinic'`+자기 `clinic_id` 방문만, 환자 기록도 자기 clinic 것만(RPC 경유).
- **연결 전 PII 차단:** 대조는 RPC 내부만. 제공 항목은 동의 후에만 병원 노출.
- **쓰기 단일 진입점:** 회원 소유 `diaries` 생성은 `clinic_*` SECURITY DEFINER RPC만(diaries.relforcerowsecurity=false → owner가 RLS 우회, 기존 write RPC와 동일 메커니즘). authenticated 직접 INSERT는 RLS로 self-owned만 허용 → 병원이 회원 행 위조 불가. 선택적 방어: INSERT 정책에 `role='clinic'` 자기 diaries 생성 차단(정정 §F-M1).
- **위협 모델:** 입력 권한 `is_clinic`(힐하우스 5계정) → 임의 공격자 불가. handle 준공개 + 이름·생일 대조 + **동의 게이트** + rate-limit. 잔여는 회원 알림·삭제·해제로 정정.
