## 1. 배경 진단 (v1 §1·§2 요약)

- 화면이 "로그인" 프레이밍 + 가입 안내가 최하단 1줄 → 신규 사용자에게 가입 경로가 안 보임.
- SNS 표준: OAuth 중심 앱은 "○○로 시작하기" **단일 진입**(가입=로그인 겸용, 시스템이 판별) + 명확한 카피. 수단 노출은 2~3개로 제한하고 위계를 분명히.
- (v1 에서 제안한 이메일 접힘/병원 라벨은 원장 결정으로 철회 — 이메일은 구분 없는 일반 수단으로 제공.)

   [ Apple로 시작하기  ]

   ─────── 또는 이메일로 ───────
   [ 이메일 ]
   [ 비밀번호 ]
   [ 로그인 ]
   이메일로 가입하기 · 비밀번호를 잊으셨나요?   ← Phase 2 링크(Phase 1 에선 미노출)

## 3. Phase 2 — 이메일 회원가입 + 비밀번호 재설정 (신설, 중형)

> 원장 결정에 따른 신설. 현재 이메일 계정은 관리자가 만든 운영 계정뿐이라, 일반 사용자가 이메일 로그인을 쓰려면 **가입·재설정 흐름이 있어야** 합니다. 기존 인프라를 최대한 재사용하는 설계입니다.

### 3.1. 흐름 설계 (기존 경로 재사용이 핵심)

```
[이메일로 가입하기] → 가입 폼(이메일·비밀번호·비밀번호 확인)
  → supabase.auth.signUp({ email, password, options:{ emailRedirectTo: /auth/callback } })
  → "확인 메일을 보냈어요" 안내 화면
  → 사용자가 메일의 링크 클릭
  → /auth/callback?token_hash=…&type=signup   ← 기존 콜백의 verifyOtp 분기가 이미 처리(재사용)
  → 세션 발급 → (기존과 동일) 약관 게이트(/signup) → 온보딩 → 홈
```

- **약관·온보딩·중복계정 방어 전부 기존 재사용**: profiles 자동 생성 트리거, terms 게이트, birthdate 온보딩 게이트, 동일 이메일 타 provider 중복 감지(`find_other_auth_user_by_email` — OAuth 로 이미 가입한 이메일로 이메일 가입 시 /login/conflict 안내)가 콜백에서 그대로 동작.
- **비밀번호 재설정**: `resetPasswordForEmail(email, { redirectTo: /auth/reset-password })` + 신규 페이지 `/auth/reset-password`(새 비밀번호 입력 → `updateUser({ password })`). 이건 기존 운영 계정(원장님들)에게도 유용(현재는 관리자가 수동 재발급).

### 3.2. 수정 명세

| # | 항목 | 내용 |
|---|---|---|
| 1 | `/signup/email` 페이지 신규 | 이메일·비밀번호(8자+)·확인 입력 → signUp → 발송 안내 화면. 이미 가입된 이메일이면 Supabase 에러를 한글로 안내("이미 가입된 이메일이에요 — 로그인하거나 비밀번호를 재설정하세요") |
| 2 | `/auth/reset-password` 페이지 신규 + LoginForm 에 "비밀번호를 잊으셨나요?" 링크 | recovery 토큰 세션에서 updateUser. 완료 후 홈 |
| 3 | LoginForm | 이메일 폼 하단에 "이메일로 가입하기" · "비밀번호를 잊으셨나요?" 링크 2개 |
| 4 | `/auth/callback` | 변경 최소 — token_hash(type=signup/recovery) 분기 동작 확인·recovery 는 reset 페이지로 라우팅 분기 1건 |
| 5 | 문서 | PRD §4.3(인증 수단에 이메일 추가)·SECURITY(비밀번호 정책)·CHANGELOG |

### 3.3. 운영 전제 (코드 밖 — 반드시 선행)

| # | 항목 | 왜 |
|---|---|---|
| 1 | **커스텀 SMTP 연결(필수)** | Supabase 기본 발신은 시간당 수 통 수준 제한 + 발신자가 supabase 도메인 → 확인 메일·재설정 메일 운영 불가. Resend/AWS SES 등 + `no-reply@pibutenten.kr`(DNS SPF/DKIM 등록) 필요 |
| 2 | Supabase Auth 설정 | Email provider 의 신규 가입 허용 상태 확인, 확인 메일 필수(Confirm email ON — 스팸 가입 방지), 메일 템플릿 한글화 |
| 3 | 스팸·남용 방어 | signUp·reset 호출에 기존 rate-limit lib 적용(IP 기준) |

- 규모: 페이지 2개 신규 + 폼 수정 + 설정. **1~2일 + SMTP/DNS 작업**. 리스크: 중간(인증 신규 경로 — 기존 OAuth 흐름은 무변경이라 회귀 없음, 신규 경로만 검증하면 됨).
