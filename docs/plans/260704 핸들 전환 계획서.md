# @핸들 네임스페이스 전환 — 실행계획서 (260704)

> **상태: 원장 승인 대기(착수 지시 시 이 문서 기준).**
> **착수 시 현행 코드로 재검증** — 아래 파일:라인 번호는 조사 시점(HEAD 기준) 값으로, 이후 커밋으로 줄 이동 가능. 각 단계 착수 전 해당 파일을 다시 열어 라인·패턴을 재확인한다.
>
> 이 문서가 이 전환의 단일 계획 SSOT. 회원 프로필·글 URL 을 `/{handle}` → `/@{handle}` 네임스페이스로 옮기는 작업의 조사·단계·위험·검증·작업량을 담는다. 구현은 원장 착수 지시 후 CLAUDE.md §3 자동실행 룰로 진행.

## 0. 핵심 원칙 (먼저 못 박음 — 재논의 금지)

디렉터 확정 사항. 아래는 전환 전 과정에서 불변 전제다.

1. **네임스페이스 이동.** 회원 프로필 `/{handle}` → `/@{handle}`, 회원 글 `/{handle}/{shortcode}` → `/@{handle}/{shortcode}`. `@` 는 URL 표시·라우팅 접두일 뿐 **DB 저장 handle 값에는 포함하지 않는다**(profiles.handle 은 `@` 없는 값 유지, 형식 CHECK `^[a-z0-9][a-z0-9-]{1,28}[a-z0-9]$` 그대로).
2. **옛 주소 → 새 주소 301.** ⚠ `card_public_url` DB 트리거가 알림 url 을 **영구 저장**(사후 변동 없음)하므로 옛 형식 `/{handle}/{shortcode}` 301 은 사실상 **무기한 유지**. "전환 정착 후 301 제거"를 전제하지 않는다.
3. **원장(의사) URL 은 `/doctors/{slug}/...` 키워드 주소 유지.** 단 원장 `@핸들` 입구는 열되(`/@{doctorHandle}`) `/doctors/{slug}` 로 301(현행 `/{handle}` → `/doctors/{slug}` 308 로직의 @ 버전).
4. **TS `card-url.ts::getQaUrl` ↔ SQL `card_public_url` 은 한 커밋 동시 갱신**(CLAUDE.md §5 dual-SSOT 페어). 프로필 URL(`profileUrl` ↔ `get_notifications`/트리거의 handle 소비)도 같은 원칙.
5. **예약어(reserved_handles) 축소는 전환 완료·301 정착 후 별도 단계.** 이 계획서 범위 밖(부록 B 에 근거만 기록).
6. **공개 초기인 지금이 전환 적기** — 회원 프로필/글이 전부 noindex + sitemap 미포함이라 색인 재구축 비용이 최소(§4 위험에서 재확인).

### 1.1. 라우트 트리 (handle 을 동적 세그먼트로 수신)

- **`@` 수용 가능성(검증된 사실 재확인):** `@` 는 URL pchar 로 유효하므로 기존 `[handle]` 동적 세그먼트가 `params.handle = "@foo"` 로 수신 가능 — **별도 라우트 신설 불요**. 단 세그먼트 값이 `@foo` 로 들어오므로 **파싱 시 `@` 를 벗겨 DB 조회**해야 한다(§2 S1). ⚠ **착수 시 실검증 필수**: 로컬 dev + production 프리뷰에서 `/@testhandle` 이 실제로 `[handle]` 로 라우팅되고 `params.handle` 값에 `@` 가 포함되는지(디코딩 여부 포함) 확인 후 다음 단계 진행.

## 2. 단계 설계 (의존 순서)

각 단계 = 1커밋(명시된 경우 분할). CLAUDE.md §3 자동실행 룰(코드 → 마이그 → build+tsc → code-reviewer → commit/push → 문서) 적용. **선행 조건: 2026-07-04 이중검증 문서의 R1-2(M-7 앵커 카드 URL 통합) 완료 후 착수**(card-url.ts 사본 정리가 선행돼야 수렴 대상이 명확).

### S1 — 라우트 수용 + 파싱 (@ 벗기기) 【중】
**목표:** `/@{handle}` 진입이 정상 렌더되게(옛 `/{handle}` 도 아직 동작 — 이중 수용 구간).
- 파일: `[handle]/page.tsx`(108/119/211 정규화), `[handle]/[shortcode]/page.tsx`(39/52 정규화), `hidden-card.ts`(정규화 비교), `route-class.ts`(검증 케이스), `GlobalChrome.tsx:65`(HANDLE_RE `@` 확장).
- `normalizeHandle(seg)` 유틸 신규(`@` 벗기고 반환). 검증 정규식은 벗긴 값 기준.
- **이 단계에서 `@` 없는 옛 진입은 아직 301 안 함**(S3 에서). 이중 수용으로 회귀 격리.
- 마이그: 없음.
- 검증: `/@{실제handle}` 프로필·글 200 + 셸 승격(footer 미중복) 확인. `/{handle}`(구) 도 여전히 200. `@` 라우팅 실검증(1.1).
- 롤백: 커밋 revert(옛 경로만 살아 있어 무영향).

### S2 — 링크 생성 SSOT 전환 (@ 접두) 【대】
**목표:** 앱이 생성하는 **모든 새 링크**가 `/@{handle}` 형식.
- **S2a (헬퍼 SSOT):** `card-url.ts::getQaUrl`(41) + `profileUrl`(97) 을 `@` 접두로. `profileUrl` 부활(R6-1 보류 해제).

### S3 — 옛 주소 → 새 주소 301 【중】
**목표:** `/{handle}`·`/{handle}/{shortcode}`(구) → `/@{handle}`·`/@{handle}/{shortcode}` 영구 301.
- **설계 결정 지점(착수 시 택1, §3 위험 참조):**
  - **(가) 미들웨어 방식**(권장): `middleware.ts` 에 report 308 패턴(`:201`)과 동형으로 첫 세그 판정 — 비예약(`!RESERVED_FIRST_SEGMENT.has`) + 비-@ 시작 + handle 정규식 매칭이면 `/@{seg}` 로 301. `@` 접두는 재진입 시 매칭 제외(무한루프 차단). 2세그(글)도 동일. **장점:** page 렌더 전 처리, 정적 라우트 우선순위 영향 없음.
  - **(나) 페이지 방식:** `[handle]/page.tsx`·`[shortcode]/page.tsx` 진입 시 세그에 `@` 없으면 `permanentRedirect(/@...)`. **단점:** 의사 slug redirect(211)·hidden·notFound 와 분기 순서 얽힘.
  - `next.config.ts::redirects()` **블랭킷 룰 불가** — `/:path*` 소스가 모든 top-level 정적 라우트와 충돌, 예약어 제외를 source 패턴으로 표현 못 함(has-조건으로도 부적합). config 방식 배제.
- 마이그: 없음(코드 리다이렉트).
- 검증: 구 URL 6유형(프로필/글/의사프로필경유/hidden/notFound/대소문자) → 새 URL 301 확인. **무한 리다이렉트 없음**(curl -I 로 Location 1홉 확인). 카톡 공유 옛 링크·북마크 시뮬.
- 롤백: 리다이렉트 블록 revert(S1 이중 수용이 살아 있어 옛 URL 여전히 200).

### S4 — dual-SSOT SQL 동시 갱신 【소】
**목표:** DB `card_public_url` 을 `/@` 형식으로(신규 알림 url 이 새 형식 저장).
- 마이그 1개(0346±): `card_public_url` CREATE OR REPLACE — 회원글 분기 `'/@' || p.handle || '/' || c.shortcode`. **getQaUrl(S2a)과 한 커밋 그룹**(CLAUDE.md §5 페어 — 이상적으로 S2a 커밋에 마이그 동봉, 또는 직후 커밋).
- **결정 지점 — 기존 알림 url 백필:** 회원글 알림 url(옛 `/{handle}/{shortcode}`)을 `/@` 로 UPDATE 할지. **권장: 백필 O**(0298 선례 멱등 UPDATE, 순수 ASCII). 단 백필 안 해도 S3 301 이 흡수 → 미백필도 안전(디렉터 선택).
- 적용: Management API UTF-8 경로(비-ASCII 없지만 습관 유지) + 적용 후 `SELECT card_public_url()` 샘플로 `/@` 확인.
- 검증: 새 좋아요/댓글 발생 → notifications.url `/@` 저장 확인. (백필 시) 기존 회원글 알림 url `/@` 확인.
- 롤백: 이전 함수 정의로 CREATE OR REPLACE(멱등). 백필은 역UPDATE 또는 301 에 위임.

### S5 — 검증 · 색인 · 문서 【소】
- **E2E**(§4): 옛/새 URL 전 유형, 알림 클릭, 공유 링크, 셸 렌더, 캐시 반영.
- **색인:** 회원 URL 은 noindex + sitemap 미포함이라 GSC 재제출 불요. robots/sitemap 재확인(변화 없음 실증). 혹시 외부(카톡 캐시 등) 옛 링크는 301 로 흡수 — 모니터링만.
- **문서:** CHANGELOG(전환 이력) + ARCHITECTURE(URL 정책 섹션) + `next.config.ts` 상단 URL 정책 주석(28-29줄) + CLAUDE.md §5 dual-SSOT 페어 표(card_public_url·getQaUrl 행에 `@` 규칙 반영) + 새 ADR(`decisions/NNNN-handle-at-namespace.md`).

## 부록 B — 예약어 축소 근거 (별도 단계, 참고만)
`@` 네임스페이스 도입 후 회원 핸들은 `@` 접두라 top-level 정적 라우트(`/admin` 등)와 **경로가 원천 분리**된다. 따라서 0336 이 확장한 reserved_handles(라우트명 선점 방지 목적) 상당수가 불필요해진다. 단 §0-5 대로 **전환·301 정착 후** 별도 안건으로 다룬다(성급한 축소 금지 — 301 이 옛 `/{handle}` 을 여전히 해석하는 구간 동안은 예약 유지가 안전).
