# 0021 — 무료/per-IP-제한 서드파티 API는 공유 서버리스 IP로 프록시하지 않는다

- 상태: 채택 (2026-06-24)
- 맥락: 피부날씨(Open-Meteo) 위치 '전원 대치동' 장애 사후 분석

## 배경 (사고)

2026-06-24 02:28, 피부날씨 콜드로드 첫 표시 지연을 줄이려 Open-Meteo(대기질·예보) 호출을
클라이언트 직접 호출 → 서버 프록시(`/api/weather`)로 옮겼다(커밋 `77b4b59`). 좌표 2자리
반올림 + 엣지 `s-maxage=600` 공유 캐시로 같은 좌표(특히 기본값 대치동)는 거의 즉시 응답하게 한
의도였다.

같은 날 종일, 서울 사용자 다수가 피부날씨 위치가 **모두 대치동(기본값)** 으로 표시된다고 제보.
측위(geolocation)는 정상인데 사용자 좌표 날씨를 못 받아 기본값 폴백이 잔존하는 증상.

## 근본 원인

- 프록시는 **모든 사용자 요청을 Vercel 서버의 공유 egress IP** 한 곳으로 모은다.
- Open-Meteo **무료 티어는 "IP당" 한도**로 차단한다(요청 출처 IP 기준).
- Vercel egress IP는 피부텐텐 전용이 아니라 **전 세계 수많은 Vercel 앱이 공유**한다.
- 따라서 우리 트래픽이 적어도 **다른 테넌트들의 Open-Meteo 호출과 합산**으로 그 IP의 한도가
  소진되면 우리 요청까지 429/502를 받는다 — **우리 사용량과 무관하게** 터진다.
- 캐시된 기본값 좌표(대치동 — 전 사용자 공유라 캐시 히트)만 항상 성공하고, 사용자별 고유 좌표
  (캐시 미스 → 업스트림 호출 필요)는 실패 → `useWeather` 의 정밀 측위 결과가 표시되지 못하고
  1단계 대치동 필러가 잔존 → "전원 대치동".

핵심: 측위·헤더(Permissions-Policy `geolocation=(self)`)·서비스워커는 모두 정상이었다. 단일
원인은 "무료 per-IP 제한 API를 공유 서버 IP로 프록시"한 설계 결함이었다.

## 결정

**무료 또는 per-IP 제한이 있는 서드파티 API는 공유 서버리스(Vercel) IP를 통해 프록시하지
않는다.** 프록시 이전의 **클라이언트 직접 호출**(각 사용자가 자기 단말 IP로 호출 → 분산되어
개인이 한도에 닿지 않음)을 표준으로 유지한다.

복구(같은 날):
- `fetchWeather` 를 Open-Meteo 2곳 클라이언트 직접 호출로 환원.
- `src/app/api/weather/route.ts` 삭제.

## 그래도 속도가 필요하면 (대안)

원래의 콜드로드 지연 문제는 다음 중 **공유 IP 합산 한도에 노출되지 않는** 방법으로만 해결한다.

1. **클라이언트 직접 호출 + 클라 캐시 유지** (현재): `useWeather` 의 localStorage seed +
   stale-while-revalidate 로 재방문·새 탭은 즉시 표시. 첫 방문 콜드로드만 분산 직접 호출.
2. **페이로드 슬림화**: 첫 페인트엔 현재값만(=`current`), 7일 hourly 는 지연 로드.
3. **API 키 기반 유료 플랜**: 한도가 IP가 아니라 키 단위 → 서버 프록시 캐시 가능(비용 발생).
4. **자체 데이터 캐시 레이어**(예: 우리 KV에 좌표별 스냅샷 저장)로 업스트림 호출 자체를 최소화.

❌ 금지: 무료 per-IP API를 그대로 서버 프록시(공유 egress IP)로 캐싱하는 방식.

## 영향

- 콜드로드(첫 방문) 첫 표시가 프록시 캐시 히트보다 다소 느려질 수 있으나, 재방문은 클라 캐시로
  즉시. **정확도(실제 위치) > 콜드로드 속도** 우선.
- 동일 패턴(서드파티 무료 API의 서버 프록시)을 다른 기능에 도입할 때 본 ADR을 먼저 검토한다.
