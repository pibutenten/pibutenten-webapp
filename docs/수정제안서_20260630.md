## #3. /login "회원가입" 클릭 시 /signup → /login 바운스 버그

### 현상 진단 (먼저)

이건 "버그"라기보다 **설계와 링크가 어긋난 상태**입니다.

- `signup/page.tsx` L28-30: 비로그인 사용자가 `/signup` 에 들어오면 `redirect('/login')` 으로 **의도적으로** 튕깁니다. `/signup` 은 "신규 가입 폼"이 아니라 **OAuth 콜백 직후 약관 동의 게이트**이기 때문입니다(소셜 로그인 = 실제 가입).
- 그런데 `login/LoginForm.tsx` L151-159 의 "처음이신가요? 회원가입" 링크가 `/signup` 을 가리킵니다.
- 결과: 비로그인 사용자가 그 링크를 누르면 → `/signup` → (user 없음) → `/login` 으로 즉시 되돌아옵니다. 사용자 눈엔 "아무 일도 안 일어남 / 깜빡임"으로 보입니다.

즉 **진짜 가입 입구는 같은 화면 위쪽의 소셜 로그인 버튼**(L68-82, "3초만에 가입 / 로그인")인데, 링크가 엉뚱한 곳을 가리키는 게 원인입니다.

### (a) 권장 접근

"회원가입" 링크를 **소셜 로그인 섹션으로 유도**하도록 바꿉니다. 두 가지 방식 중 택일:

1. **앵커 스크롤 방식** (권장): `/signup` 링크 → 같은 페이지의 소셜 버튼 영역으로 부드럽게 스크롤 + 살짝 하이라이트. "피부텐텐은 소셜 계정으로 3초 만에 가입돼요" 안내 문구를 함께.
2. **문구 자체 교체**: "처음이신가요? 회원가입" → "위 버튼으로 3초 만에 시작하세요" 같이 **별도 페이지로 보내지 않는** 안내문으로 바꾸고 링크 제거.

소셜 로그인이 곧 가입인 제품 구조상, 별도 "회원가입 페이지"라는 개념 자체가 사용자에게 혼란입니다. 입구를 하나(소셜 버튼)로 모으는 게 정답입니다.

### (b) 대안

- **대안 1 — 이메일 회원가입 폼 신설**: `/signup` 을 진짜 가입 폼으로 만든다. → 비권장. 현재 인증 모델(OAuth 단일)을 깨고, dedup·온보딩·약관 흐름을 전부 이메일 경로로 중복 구현해야 합니다. 과한 변경입니다.
- **대안 2 — `/signup` 진입 시 `/login` 대신 `/login#social` 로 보내고 자동 스크롤**: 링크는 그대로 두고 도착지에서 처리. (a)-1 과 결과는 같으나 리다이렉트가 한 번 더 끼어 깜빡임이 남습니다.

### (c) UX·심미성·일관성 트레이드오프

- **일관성**: 입구를 소셜 버튼으로 단일화하면 `EngagementPromptDialog`(소셜 버튼 + /signup + /login), 온보딩 dedup 다이얼로그의 "기존 계정으로 로그인" 흐름과 톤이 맞습니다.
- **심미성**: "회원가입"이라는 단어를 남기면(앵커 방식) 신규 사용자가 익숙한 멘탈모델을 유지하면서도 페이지 이동 없이 처리돼, 깔끔합니다.
- **속도**: 페이지 이동·리다이렉트가 사라져 체감 즉시성이 올라갑니다.

### (d) 리스크

- 거의 없음. 단 `/signup` 자체를 **삭제하면 안 됩니다** — OAuth 콜백 후 약관 게이트로 계속 쓰입니다. 링크만 손대세요.
- `next` 파라미터(로그인 후 복귀 경로)가 소셜 버튼 경로에서도 보존되는지 확인 필요(L79 SocialLoginButtons 가 `next` 를 받는지).

## #2. 특정 시술 리포트에서 "내 후기 남기기" → 후기 폼에 그 시술 자동 선택

### 현상 진단

**후기 폼 쪽은 이미 완성돼 있습니다.** `review/new/page.tsx` L42-45:

```
const initialProcedure =
  sp.procedure && procedures.some((p) => p.value === sp.procedure)
    ? sp.procedure : undefined;
```

즉 `/review/new?procedure={시술ko}` 로 들어오면 그 시술이 자동 선택됩니다. **문제는 호출하는 쪽**입니다.

- `reports/ReportsIndexCard.tsx` L333-345: CTA가 `<Link href="/write?tab=review">내 후기 남기기</Link>` — (1) 레거시 경로 `/write?tab=review` 사용, (2) `procedure` 파라미터 없음. 카드는 L75-76 에서 `procedureKo` 를 이미 알고 있습니다.
- `reports/[procedure]/ReportsDetailView.tsx`: 상세 화면엔 **"내 후기 남기기" 버튼이 아예 없습니다.** 정작 "이 시술 리포트를 보는 중"인 화면은 상세 뷰인데, 거기에 진입 버튼이 없는 게 더 큰 공백입니다.

### (a) 권장 접근

두 곳을 함께 처리합니다.

1. **인덱스 카드 CTA 경로·파라미터 교정**: `/write?tab=review` → 캐노니컬 경로 `/review/new?procedure=${encodeURIComponent(procedureKo)}`.
2. **상세 뷰에 "내 후기 남기기" 버튼 신설**: 상세 화면 하단(또는 헤드라인 근처)에 동일 CTA를 두고 같은 파라미터로 연결. 사용자가 "지금 보고 있는 시술"에 바로 후기를 남기는 자연스러운 동선을 완성합니다.

`encodeURIComponent` 는 필수입니다(한글 시술명 ko 가 그대로 쿼리에 들어가므로).

### (b) 대안

- **대안 1 — 인덱스 카드만 수정**: 최소 변경이지만, 정작 후기 작성 의향이 가장 높은 "상세 뷰 정독 후" 진입점을 놓칩니다. 비권장.
- **대안 2 — 레거시 `/write?tab=review` 도 procedure 파라미터를 받게 확장**: 후기 폼이 이미 `/review/new` 로 캐노니컬화돼 있어, 레거시를 살려두면 SSOT가 둘로 갈립니다. PRD §4.3 의 캐노니컬 폼 방향과 어긋나므로 비권장.

### (c) UX·심미성·일관성 트레이드오프

- **일관성**: 모든 후기 진입을 `/review/new` 로 모으면 폼 경로가 단일화됩니다.
- **UX**: 시술이 미리 선택돼 있으면 입력 단계가 하나 줄어 전환율이 오릅니다. 단, 자동 선택 후에도 **사용자가 변경 가능해야** 합니다(L42-45 가 dropdown 초기값일 뿐 잠금이 아니므로 OK).
- **심미성**: 상세 뷰 버튼은 `categoryTheme.color` 를 따르게 해 시술별 색감과 통일하면 좋습니다.

### (d) 리스크

- 매우 낮음. 단 `?procedure=` 값이 후기 폼의 옵션 목록(`procedures`)에 **존재하지 않는 시술명**이면 L42-45 가 `undefined` 로 안전하게 무시하므로, 리포트 풀과 후기 폼 옵션 간 시술명 표기가 일치하는지만 확인하면 됩니다(family 롤업 차이로 인덱스엔 있는데 후기 폼 enum엔 없는 케이스 점검).

## #6. "다시 받고 싶어요" → "재시술의향" 으로 통일하고 1차 지표로

### 현상 진단

라벨이 화면마다 다릅니다.

- `reports/ReportsIndexCard.tsx` L174: **`다시 받고 싶어요`** (인덱스 카드 지표 라벨) ← 교체 대상.
- `reports/[procedure]/ReportsDetailView.tsx` L331-333: 상세 히어로는 **이미 `재시술의향`** 으로 돼 있습니다.
- `reports/ReportsIndexView.tsx` L40-41: 정렬 칩은 이미 `재시술의향 높은 순`.
- `ReportsDetailView.tsx` L369: `다시 받고 싶어 해요` — 이건 **라벨이 아니라 본문 서술문**(자연어). 라벨 통일과 별개이며, 본문은 서술형 톤을 유지하는 게 자연스럽습니다.

즉 **인덱스 카드 L174 한 곳만** 라벨이 옛 표현으로 남아 있습니다.

### (a) 권장 접근

1. **라벨 교체**: 인덱스 카드 L174 `다시 받고 싶어요` → `재시술의향`.
2. **1차 지표 격상(시각 위계)**: 카드 내에서 재시술의향 블록을 만족도·통증보다 **위/크게** 배치하거나, 강조색(브랜드색)을 부여해 "이 앱이 가장 중요하게 보는 숫자"임을 시각적으로 표현. 상세 히어로(L331-333)가 이미 `theme.color` 강조를 쓰므로 인덱스 카드도 동일 위계를 맞춥니다.
3. **본문 서술문(L369)은 유지**: 라벨은 "재시술의향", 문장은 "다시 받고 싶어 해요" — 라벨/문장 톤 분리는 의도된 패턴입니다.

### (b) 대안

- **대안 — 전부 "재시술의향"으로 기계 치환**: L369 본문까지 바꾸면 "사용자의 78%가 재시술의향이에요" 같은 어색한 문장이 됩니다. 비권장. 라벨과 서술문을 구분하세요.

### (c) UX·심미성·일관성 트레이드오프

- **일관성**: 인덱스·상세·정렬 칩 3곳의 **라벨**이 "재시술의향"으로 통일됩니다.
- **의미 전달**: "다시 받고 싶어요"는 따뜻하지만 모호하고, "재시술의향"은 데이터 지표로서 명확합니다. 리포트는 데이터 신뢰가 핵심이라 후자가 적합합니다.
- **리포트 카피 SSOT 주의**: PRD/리포트 카피 규약상 "본문 서술형은 '경험', SEO 메타는 '후기'" 같은 단어 규약이 있으니, "재시술의향"을 지표 라벨 SSOT로 못박아 두는 게 좋습니다.

### (d) 리스크

- 매우 낮음. JSON-LD·메타 디스크립션에 "다시 받고 싶어요" 문자열이 박혀 있지 않은지만 grep 으로 한 번 확인(SEO 텍스트 회귀 방지).

## #4. 온보딩 전면 개편 + 피츠패트릭 문항 추가

가장 손볼 곳이 많은 항목입니다. 6개 세부 요청을 나눠 다룹니다.

### 4-1. "(택 1)" 힌트 제거

- 현황: `OnboardingClient.tsx` L551(얼굴형), L566(피부타입) 에 `hint="택 1"`.
- 권장: 두 `hint` 속성 제거. `Section` 컴포넌트(L772-799)는 `hint` 가 없으면 자동으로 안 그리므로 안전합니다.
- 리스크: 없음.

### 4-2. 피부고민 "(복수 선택)" → "모두 고르셔도 되요"

- 현황: L581 `hint="복수 선택"`.
- 권장: `hint="모두 고르셔도 되요"`.
- **맞춤법 주의**: "되요"는 비표준입니다. 표준은 **"돼요"**. 다만 사용자께서 "되요"로 명시하셨으니, **"모두 고르셔도 돼요"** 로 교정 제안을 드리고 최종 확정을 받는 걸 권합니다(브랜드 카피 신뢰도 측면).

### 4-3. 칩 색 다양화 (현재 전부 #4CBFF2 하늘색)

#### 현상

`Chip` 컴포넌트(L809-844)는 active 시 **전부 `#4CBFF2`(브랜드 하늘색)** 단색입니다. 얼굴형·피부타입·성별·피부고민이 모두 같은 색이라 단조롭습니다.
(단, 5번 관심키워드 picker 안의 칩은 별도 인라인 버튼으로 **카테고리 색**을 이미 씁니다 — L987-1013. 이건 건드리지 않습니다.)

#### (a) 권장 접근 — "섹션별 1색" 팔레트

칩마다 무지개로 칠하면 난잡합니다. **섹션 단위로 색을 배정**해, 한 섹션 안 칩은 같은 색·다른 섹션은 다른 색으로 합니다. 브랜드 하늘색을 기준으로 한 **차분한 파스텔 4색** 제안:

| 섹션 | 활성 배경 | 활성 텍스트 | 의도 |
|---|---|---|---|
| 성별 | `#4CBFF2` (브랜드 하늘) | `#FFFFFF` | 기준색 유지 |
| 얼굴형 | `#7FD1B9` (민트그린) | `#0B3B30` | 윤곽=청량 |
| 피부타입 | `#F4A988` (살구) | `#5A2616` | 피부=따뜻 |
| 피부고민 | `#B39DDB` (라벤더) | `#311B5B` | 고민=차분 |
| 피츠패트릭(신규) | `#E9B949` (앰버) | `#4A3608` | 햇볕=톤 |

- 비활성은 현행 그대로(`#E8EAEE` 배경 / `#5C6470` 텍스트)로 통일 — 선택 전엔 중립, 선택 시에만 섹션색이 켜지는 패턴이라 "내가 고른 것"이 한눈에 보입니다.
- 텍스트 색은 **각 배경에서 대비(WCAG AA, 4.5:1)** 를 통과하도록 어두운 톤으로 지정했습니다. 흰 배경 위 파스텔에 흰 글자는 대비 미달이므로 활성 텍스트는 **진한 동계열**을 권합니다.

### 5-1. "최신순" 기본 정렬 추가

#### (a) 권장 접근

- **새 정렬키 `recent` 를 SORTS 맨 앞에 추가하고 기본값으로**: L38-43 배열 선두에 `{ key: "recent", label: "최신순" }`, L57 `useState("recent")`.
- **"최신순" 정의(가중 점수)**: 요청대로 단순 시간순이 아니라 **최근 활동 + 후기 수 + 재시술의향**을 가중합한 점수:
  ```
  recencyScore =
      w1 * 정규화(최근 업데이트 시각)   // 신선도
    + w2 * 정규화(후기 수)              // 활성도
    + w3 * 정규화(재시술의향 yes 비율)  // 신뢰도
  ```
  가중치 예시 `w1=0.5, w2=0.3, w3=0.2` (활동성 우선). 이 점수는 **서버(page.tsx loadPool)** 에서 계산해 내려주는 게 정석입니다 — 클라가 "최근 업데이트 시각"을 모르기 때문(현재 풀에 `updatedAt` 류가 있는지 확인 필요. 없으면 RPC/쿼리에 추가).
- **데이터 의존성**: "최근 업데이트"를 알려면 시술별 **마지막 후기 작성 시각**이 풀에 있어야 합니다. 현재 `ProcedureReport` 에 그 필드가 없으면 집계 쿼리에 `max(published_at)` 추가 필요.

#### (b) 대안

- **대안 — 순수 시간순(max published_at desc)**: 구현 단순. 단 "후기 1건짜리 신상 시술"이 최상단을 차지해 품질 낮은 리포트가 노출됩니다. 요청의 "가중" 취지와 어긋남. 비권장.

#### (c) 트레이드오프

### 5-2. 스크롤 새로고침 시 약한 랜덤 셔플 + "피드 같은" 느낌

#### 셔플의 위험 (먼저 — 사용자께서 우려하신 부분)

"스크롤하면 새로운 게 나오는" 느낌은 **무한스크롤(새 항목 append)** 과 **순서 셔플(기존 항목 재배열)** 이 전혀 다릅니다. 둘을 섞으면 위험합니다.

- **방향 감각 상실**: 사용자가 본 카드가 새로고침마다 위치를 바꾸면 "방금 그거 어디 갔지?"가 됩니다. 리포트는 정보 탐색이지 무한 소비 피드가 아니라, 셔플이 **탐색을 방해**합니다.
- **SEO·하이드레이션 충돌**: 매 요청 순서가 바뀌면(서버 셔플) 크롤러가 보는 ItemList 순서가 불안정하고, **SSR/CSR 순서 불일치 → 하이드레이션 깨짐** 위험. 현재 page.tsx 가 헤드라인만 랜덤이고 **순서는 고정(count desc)** 인 건 이 때문입니다.
- **정렬 칩과의 모순**: 사용자가 "만족도 높은 순"을 골랐는데 셔플이 끼면 정렬이 거짓말이 됩니다.

#### (a) 권장 접근 — "셔플" 대신 "결정적 약한 회전(seeded jitter)"

완전 랜덤 대신, **상위권은 고정하고 동점·근접 구간만 약하게 섞는** 방식을 권합니다.

1. **틱 시드 회전**: "최신순" 점수가 **비슷한(동점 밴드)** 카드끼리만 `seed = floor(now / 30분)` 기반으로 순서를 회전. 점수 차가 큰 카드는 자리 유지 → 상위 신뢰 리포트는 안정, 중위권만 신선도 부여.
2. **pull-to-refresh 에서만 갱신**: 무한스크롤 중 위치 변동 금지. `FeedView` 처럼 **당겨서 새로고침할 때만** 새 시드로 재정렬(router.refresh). 스크롤 도중엔 절대 안 흔들림.
3. **정렬 칩이 "count/만족도/통증"일 땐 셔플 OFF**: 셔플은 **"최신순"에서만**. 명시적 정렬을 고른 사용자에겐 정직한 순서.

#### (b) 대안

- **대안 1 — 진짜 무한스크롤 도입(FeedView 패턴 이식)**: 풀을 페이지네이션해 `/api/reports?cursor=` 로 20개씩 append. "스크롤하면 새 게 나옴"을 **순서 변경 없이** 달성 — 가장 안전하게 요청 취지를 만족. 단 리포트 풀은 현재 전체가 작을 수 있어(자격 N≥4 시술만) 무한스크롤이 과할 수 있음. **풀 크기 확인 후 결정**.
- **대안 2 — 클라 셔플(useState 시드)**: 서버 부담 0, SEO 영향 0(크롤러는 SSR 초기 순서만 봄). 단 새로고침 때마다 시드 바뀌면 위 "방향 감각" 문제 잔존. pull-to-refresh 시에만 시드 갱신하면 완화.
- **대안 3 — 셔플 포기, "최신순" 정렬만**: 가장 안전. "피드 같은 신선함"은 최신순 가중치(최근 업데이트 반영)만으로도 상당 부분 달성됩니다. 셔플 리스크가 부담되면 이 안.

#### (c) UX·심미성·일관성 트레이드오프

- **"피드 느낌"의 본질**: 사용자가 원하는 건 "올 때마다 새로움"입니다. 이는 **(1) 최신순 가중 + (2) pull-to-refresh 시에만 회전** 으로 안전하게 달성 가능하고, **스크롤 중 흔들림** 없이 신뢰를 유지합니다.
- **일관성**: pull-to-refresh→router.refresh 는 이미 FeedView 가 쓰는 앱 표준 동작이라, 리포트에 도입해도 학습 비용 0.
- **심미성**: 회전은 "동점 밴드 내"로 제한해 레이아웃이 미세하게만 변해 차분합니다.

#### (d) 리스크

- **높음**.
  - **하이드레이션**: 서버 셔플 시 SSR/CSR 불일치 → 반드시 **시드를 서버에서 확정해 prop으로 내려** SSR/CSR 동일 순서 보장(현재 헤드라인도 같은 이유로 서버 확정). 클라 재랜덤 금지.
  - **SEO**: ItemList(JSON-LD) 순서가 흔들리면 색인 신호가 약해질 수 있음. JSON-LD 의 itemListElement 순서는 **셔플과 무관하게 안정 키(count desc 등)로 고정** 권장 — 화면 순서와 JSON-LD 순서를 분리.
  - **정렬 칩 상태**: 셔플이 사용자의 명시 정렬을 덮지 않도록 가드.
  - SEO검수관(schema-auditor) 검토 필수 — JSON-LD·canonical·force-dynamic 영향.

## #1. 비로그인 하드월 강화 + 모바일 앱 설치 유도

**6건 중 리스크가 가장 큽니다.** 트래픽·SEO·전환에 직결되므로, 제안만 드리고 적용 전 별도 의사결정을 강력히 권합니다.

### 현상 진단

현재는 **소프트월**입니다.

- `lib/engagement-score.ts`: 세션 동안 행동 점수를 누적(카드열람 +2, 펼침 +2, 동영상 +3, 검색 +3, 이동 +1, 체류 +3~5)해 **THRESHOLD=15** 도달 시 프롬프트. 세션당 1회, 7일 dismiss.
- `components/EngagementPromptDialog.tsx`: 모달(소셜 버튼 + /signup + /login). **차단이 아니라 권유** — 닫으면 계속 봅니다.
- `app/app/AppStoreRedirect.tsx`: `/app` 진입 시 UA로 iOS/Android 판별 → 스토어 리다이렉트. 크롤러는 JS 미실행이라 OG 미리보기 안전.

요청은 이를 **Instagram/Threads 식 하드월**(비로그인 다음 페이지 이동 차단 + 즉시 로그인 유도)로 올리는 것입니다.

### "하드월 vs SEO·이탈" 트레이드오프 (핵심)

이게 이 항목의 본질입니다. **피부텐텐의 트래픽 구조**를 먼저 봐야 합니다.

- 리포트·Q&A 는 **force-dynamic + index 허용**으로 **검색 유입(SEO/GEO)** 을 노리는 콘텐츠입니다. 즉 **비로그인 방문자 = 검색에서 온 신규 = 성장 엔진**입니다.
- Instagram 식 하드월(비로그인은 1~2개 보고 차단)을 그대로 이식하면:
  - **단기**: 로그인 전환율↑ (강제하니까)
  - **장기**: **검색 이탈률 급등 → SEO 랭킹 하락 → 신규 유입 자체가 마름**. 구글은 "클릭 후 즉시 벽에 막힘(cloaking/intrusive interstitial)"을 페널티합니다. 모바일 인터스티셜은 **검색 순위 직접 감점** 요인입니다.
  - 인스타가 하드월을 버티는 건 **이미 거대한 브랜드 직접 유입**이 있어서입니다. 성장기 커뮤니티가 같은 전략을 쓰면 **유입 파이프가 끊깁니다.**

**결론**: "Instagram 식 즉시 하드월"을 **검색 유입 콘텐츠(리포트/Q&A 본문)에 그대로 적용하면 안 됩니다.** 대신 **점진적·맥락적 게이팅(metered paywall 패턴)** 이 정답입니다 — NYT·Medium·당근 등이 쓰는 방식.

### (a) 권장 접근 — "미터드 게이트" (강도 ↑, SEO 보존)

현재 소프트월을 **유지·강화**하되, 차단 시점을 **요청대로 콘텐츠 소비량 기준**으로 명확히 합니다.

1. **카운터 기반 게이트**: 세션(또는 로컬) 카운터로
   - **리포트 상세 2번째 열람 시도** → 가입 유도(요청: 1개 본 뒤 2번째에 요구)
   - **Q&A 5~6개 정독** → 가입 유도
   를 트리거. 현재 `engagement-score` 의 추상 점수 대신, **"리포트 N회/Q&A N회"라는 사용자가 납득 가능한 명시 카운터**를 추가하는 게 요청에 더 맞습니다.
2. **차단 강도는 "소프트 하드월"**: 모달을 **dismiss 가능하되, 다음 콘텐츠는 흐림(blur) + 가입 시 해제** 형태로. 완전 차단(빈 화면)이 아니라 "맛보기 후 벽"이라 검색 페널티·이탈을 줄입니다.
3. **첫 진입(검색 유입 1개)은 항상 풀 노출**: 크롤러·신규의 첫 페이지는 절대 막지 않음 → SEO 안전. 벽은 **2번째 행동부터.**
4. **모바일 앱 유도 분리**: 데스크탑=가입 유도, **모바일=앱 설치 유도**(`https://pibutenten.kr/app` → AppStoreRedirect 로 스토어 분기). 단 앱 유도 배너도 **인터스티셜 전면 차단은 금지**(구글 모바일 인터스티셜 페널티). 하단 고정 배너·일부 모달 정도로.

### (b) 대안

- **대안 1 — 진짜 인스타식 하드월(2개 후 전면 차단)**: 요청에 가장 충실. 전환율 단기 최대. **그러나 SEO·신규 유입 붕괴 리스크 최대.** 검색 트래픽 비중이 낮고 앱/직접 유입이 주력일 때만 고려.
- **대안 2 — 로그인 없이도 전부 열람, 작성/상호작용만 게이트**: 가장 SEO 친화적. 단 "로그인 유도"라는 요청 목표(가입 전환)엔 약함.
- **대안 3 — A/B 테스트**: 일부 트래픽에만 하드월을 켜고 **전환·이탈·SEO 동시 측정**. 데이터로 결정. **가장 안전한 실전 접근** — 직감으로 전면 적용하지 말고 이걸 권합니다.

### (c) UX·심미성·일관성 트레이드오프

- **이탈 vs 전환**: 벽이 셀수록 전환↑·이탈↑. "맛보기(1개) → 흐림 벽(2번째)"이 균형점입니다.
- **앱 유도 위치**: 모바일 웹 사용자에게 매 페이지 앱 모달을 띄우면 짜증·이탈. **세션 1회**, 또는 **2번째 게이트 시점에 "앱에서 더 편하게" 한 번**만.
- **일관성**: 게이트 모달은 기존 `EngagementPromptDialog` 톤(소셜 버튼 우선)을 재사용해 학습 비용 0.
- **심미성**: blur 게이트는 콘텐츠를 "있긴 한데 가려짐"으로 보여줘 호기심을 자극(빈 벽보다 전환↑).

### (d) 리스크 (가장 신중히)

- **SEO 페널티(치명)**: 구글 "intrusive interstitial" 가이드라인 위반 시 모바일 검색 순위 직접 감점. 첫 페이지 풀 노출 + 인터스티셜 회피로 방어. **SEO검수관 필수 검토.**
- **클로킹 의심**: 크롤러엔 풀, 사용자엔 벽 — 이를 UA로 분기하면 **클로킹(구글 제재)**. 반드시 **"첫 N개는 모두에게 풀, 그 다음부터 모두에게 벽"** 으로(크롤러 특별대우 금지). 크롤러는 자연히 N=1만 보고 끝나므로 색인엔 영향 없고 클로킹도 아님.
- **앱스토어 정책**: 웹에서 강제로 스토어로 튕기는 패턴은 무방하나, **iOS Safari 에서 앱 설치 강요 배너**는 Smart App Banner 표준을 쓰는 게 안전.
- **전환 데이터 부재**: 현재 소프트월 전환율 baseline 없이 하드월로 직행하면 효과를 모름 → A/B 또는 단계적 강화 권장.
- **법적**: 비로그인 차단 자체는 문제없으나, 동의 전 PII 수집 금지 원칙은 유지.
