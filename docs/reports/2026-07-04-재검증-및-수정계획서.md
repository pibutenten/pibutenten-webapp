## 2. 수정 계획서

> 원칙: **위험도·전파범위 순**. 같은 계층(DB/GRANT/함수)은 한 마이그레이션으로 묶어 회귀 테스트를 한 번에. 되돌릴 수 있게 단계마다 커밋.

### Phase 1 — 보안 배치 (C-1 + H-1 + H-2) 🔴 최우선

이 셋은 전부 DB 계층(RLS 함수·GRANT·익명화 RPC)이라 **한 마이그레이션 + 한 번의 회귀 테스트**로 처리. **이 Phase 가 서브에이전트 2명 검토의 핵심 대상.**

#### 1a. C-1 — `current_active_profile_id()` 에 묶음 검증 추가
- **방법**: 함수가 헤더 UUID 를 반환하기 전에, 그 프로필이 호출자 본인 묶음(`id = auth.uid()` 또는 `auth_user_id = auth.uid()`)인지 `EXISTS` 로 확인. 아니면 `NULL` 반환(호출측 `COALESCE` 가 `auth.uid()`=본인으로 폴백 → 공격자는 자기 자신으로만 행동 가능).
- **정확히 `is_admin()`/`current_doctor_id()` 와 같은 게이트** (그 둘은 이미 이 검증을 함 — 검증된 패턴).
- **전파**: 이 함수를 쓰는 **31개 정책이 자동으로 동시에** 안전해짐. 정책 자체는 손대지 않음.
- **성능**: `profiles.id`(PK) + `profiles_auth_user_id_idx`(존재 확인함) 인덱스로 EXISTS 는 O(1). 함수는 `STABLE` 이라 쿼리 내 캐시됨.
- **되돌리기**: 함수 이전 정의로 `CREATE OR REPLACE` (원본 백업 첨부).
- **회귀 테스트**: (1) 정상 명함 전환(묶음 9개) 여전히 동작, (2) 위조 헤더로 남의 diaries/notifications SELECT·INSERT 거부, (3) 본인 데이터는 정상.

#### 1b. H-1 — 회원 PII 컬럼 잠금 + 안전 조회 경로
- **방법 (2단계)**:
  1. `REVOKE SELECT (birthdate, contact_email, gender, face_shape, skin_type, skin_concerns, interested_procedures, fitzpatrick) ON profiles FROM authenticated;` — anon 은 이미 0325 로 차단됨, 이제 authenticated 도 차단.
  2. **본인 PII 를 읽어야 하는 곳**을 위해 `SECURITY DEFINER` RPC `get_profile_private(target uuid)` 신설: target 이 호출자 묶음이면 원본 PII 반환, 아니면 `field_visibility` 로 걸러 공개 필드만 반환. (타인 프로필의 공개 필드 노출은 **서버에서** 강제)
- **⚠️ 최대 위험 지점 (반드시 검토)**: PII 컬럼을 읽는 코드가 **약 23개 파일**. GRANT 를 회수하면 **본인이 자기 정보를 읽는 것까지** 막혀 온보딩·설정·마이페이지가 깨질 수 있음. 서버 라우트가 어떤 롤(authenticated JWT vs service_role)로 읽는지 파일별 확인 필수:
  - 본인 PII 읽기(깨지면 안 됨): `settings/profile/ProfileEditClient`, `onboarding/*`, `my/page`, `review/new/page`, `signup/*`, `identity-server`, `middleware`
  - 타인 PII 읽기(막아야 함): `[handle]/page.tsx`, `ProfileView`
  - 서버 라우트(service_role 이면 무영향): `api/articles`, `api/reviews`, `api/comments`, `api/visits`, `auth/callback`, `naver/callback`
- **전략 결정 필요**: service_role 로 읽는 서버 경로는 GRANT 무관하게 계속 동작. authenticated JWT 로 own-PII 를 읽는 클라·서버 컴포넌트만 RPC 로 전환. **이 분류가 정확해야 안 깨짐** → 검토 위임.
- **되돌리기**: `GRANT SELECT (...) ON profiles TO authenticated;` 재부여.
- **회귀 테스트**: 온보딩/설정/마이페이지에서 본인 생년월일·피부타입 정상 표시 + 저장, 타인 프로필에서 비공개 필드 미노출(REST 직접 호출로도).

#### 1c. H-2 — 탈퇴 익명화를 묶음 전체로 복원
- **방법**: `anonymize_user_content_before_delete()` 의 `UPDATE ... WHERE id = v_target` 를 **묶음 전체 루프**(`FOR ... IN SELECT id FROM profiles WHERE id = v_uid OR auth_user_id = v_uid`)로 복원(0109 원설계 방식).
- **⚠️ 제품 판단 필요 (검토 대상)**: 의사 명함까지 익명화되면 그 의사의 공개 Q&A 카드 작성자명이 `(탈퇴한 사용자)`로 바뀜. 콘텐츠 본문은 보존되나 **의사 실명 표기 정책**과 충돌 가능 → 원장 확인.
- **병행**: 트랜잭션 갭 — RPC 안에서 auth 삭제까지 하거나, 라우트에서 실패 시 재시도 안전화(auth_user_id NULL 처리 순서 조정).
- **소급**: 이미 탈퇴한 계정 중 `auth_user_id IS NULL AND deleted_at IS NULL` 고아 프로필 점검·소급 익명화.
- **되돌리기**: 익명화는 비가역(PII 삭제)이므로 **실행 전 백업 필수**. 함수 정의는 되돌릴 수 있음.

**Phase 1 산출물**: 마이그 `0329_active_profile_bundle_check.sql`(1a) + `0330_profiles_pii_lockdown.sql`(1b) + `0331_anonymize_bundle_restore.sql`(1c) + 코드 변경(1b RPC 소비처) + 회귀 테스트.

## 4. 서브에이전트 2명 회귀 검토 결과 (2026-07-04) + 독립 재검증 진행 중

Phase 1 을 실제로 실행하면 무엇이 깨지는지 독립 검토자 2명(앱/코드 관점 · DB 관점)에게 위임한 결과. **검토자 주장 자체를 다시 독립 에이전트로 사실검증 중** — 확정 전.

### 검토로 드러난 핵심 (계획 대비 보완 필요)
- **1a(C-1)는 계획대로 안전.** 성능 영향 무시 가능(`profiles` PK + `auth_user_id` 인덱스, STABLE 캐싱). `auth.uid()=base id` 가정도 실측 확인(base 176 + 서브 10 정상 매핑). 함수 사용 정책은 30개(계획서 31개는 근사).
- **1b(H-1)는 계획보다 위험 — 배포 순서 필수.** 제 원 계획의 "서버 라우트=service_role 무영향" 분류가 **틀림**. `createSupabaseServerClient()`는 authenticated(사용자 JWT) 롤. REVOKE 하면:
  - `identity-server.ts:96`(birthdate SELECT) → `resolveActiveIdentity` null → **모든 로그인 API 401**(blocker)
  - `middleware.ts` 온보딩 게이트가 에러 시 `return response`로 스킵 → **미온보딩 사용자 전 라우트 접근**(보안 역행 blocker)
  - `onboarding/page`·`my/page`·`[handle]/page`(본인·타인)·`admin/users`·`today/page` 깨짐
  - PostgreSQL: 컬럼 GRANT 회수 시 그 컬럼 포함 SELECT **전체가 42501 실패**(컬럼만 누락 아님)
  - → **반드시 "코드(안전 RPC 전환) 먼저 배포 → 그다음 REVOKE"** 순서. 한 마이그레이션에 몰면 안 됨.
- **1c(H-2)에 숨은 blocker.** 묶음 루프로 바꾸면 현재 `v_mask`가 루프 밖에서 1회 계산 → **모든 묶음 프로필 handle 이 동일값 → `idx_profiles_handle_unique` 위반 → 탈퇴 트랜잭션 전체 롤백(탈퇴 불가)**. 루프 안에서 `r.id` 기준으로 mask 를 각각 생성해야 함. + **1a 와 반드시 같이 배포**(안 그러면 재시도 교착). + 원장 결정대로 의사 명함 제외.
- **트랜잭션 갭**: 익명화 성공 후 deleteUser 실패 시 "로그인은 되나 아무것도 못 하는" 유령 상태. 재시도 안전화 필요.

→ 이 보완들을 반영한 **Phase 1 최종본(배포 단계 A→B 분리)**은 독립 재검증 통과 후 확정.

### 독립 사실검증 결과 (2026-07-04, 검증자 2명 — 검토자와 별개)
검토자 주장을 정답으로 주지 않고 처음부터 재현. **결론: 검토자 핵심 발견 전부 [확인], 서술 부정확 2건만 교정(결론 불변).**
- ✅ C-1 함수가 헤더를 무검증 반환(현재 profiles 조회조차 안 함) → 31개 정책 노출. 강화는 안전(STABLE·인덱스, base 176/서브 10/탈퇴 2 실측, 정상 사용자 차단 경로 없음).
- ✅ `identity-server.ts:96`(birthdate SELECT)가 **로그인 API 전체의 단일 실패점** — REVOKE 시 모든 인증 API 401. (검토자 6개 파일 목록보다 이 헬퍼가 1순위)
- ✅ middleware **fail-open 두 경로**(candidate + base fallback) 모두 게이트 스킵 → 온보딩 우회 보안역행.
- ✅ **handle unique blocker 실측 재현** — 묶음 9개(최대 3명, 19프로필)에서 순진한 루프는 동일 handle 충돌로 탈퇴 롤백. 루프 안 `r.id`별 mask 필수.
- ✅ 컬럼 REVOKE → 그 컬럼 포함 SELECT 전체 42501(anon 롤로 실측 재현). "코드 먼저 → REVOKE 나중" 순서 타당.
- ✅ card_likes/site_visits 정정 **부작용 없음**(소비 경로 전부 `.eq(profile_id)` 본인 한정 또는 SECURITY DEFINER 트리거/RPC 경유).
- 🔧 교정1(사소): C4 컬럼범위 서술 과장 — admin/users는 2컬럼만 읽음(파일이 깨진다는 결론은 유효). 교정2(사소): C-1 함수는 지금 profiles 조회 안 하므로 EXISTS는 "O(1) 유지"가 아니라 신규 조회 1회 추가(비용은 무시가능).
- ➕ 추가발견: 탈퇴 시 sub 명함 PII 잔존은 **이미 현실**(FK `ON DELETE SET NULL`). 소급 익명화 스크립트도 id별 mask 필수(공통 mask면 충돌 재현).

## 5. Phase 1 최종 확정안 (독립검증 통과 — 2단계 배포)

핵심 통찰: **C-1·H-2·자잘2건은 GRANT 회수가 없어 앱을 안 멈춘다 → 먼저 빠르게 배포해 치명 취약점을 즉시 닫는다.** H-1(PII 권한회수)만 배포 순서를 엄격히 지켜야 하므로 뒤로 분리.

### Phase 1-A — 빠른 안전 배치 (치명 즉시 해소, 회귀검증 완료)
GRANT 회수 없음 → 어느 시점에도 앱 정상. 한 배포로:
1. **C-1**: `current_active_profile_id()` 에 묶음 검증 EXISTS 추가(마이그 `0329`). → 31개 정책 동시 방어.
2. **H-2**: `anonymize_user_content_before_delete()` 묶음 루프 복원(마이그 `0330`) — **(a) 루프 안 `r.id`별 mask 생성**(handle 충돌 방지), **(b) 의사 명함 제외**(`role<>'doctor' AND doctor_id IS NULL` — 원장 결정), C-1 과 같은 배포(교착 방지).
3. **탈퇴 트랜잭션 갭**: `/api/me/delete` 재시도 안전화(auth_user_id NULL 이후 재시도 시 base로 자연 폴백 — 검증됨). 필요 시 순서 조정.
4. **자잘 보안 2건**: `card_likes` SELECT 를 본인+집계안전 형태로 좁힘, `site_visits` anon INSERT 에 형태 제약(정상 방문 적재 안 깨지게, 부작용 없음 검증됨)(마이그 `0331`).
5. **소급**: `auth_user_id IS NULL AND deleted_at IS NULL` 고아 프로필 점검(현재 0건, 예방). 소급 스크립트도 id별 mask.
- **회귀 테스트**: 명함 전환(9묶음)·좋아요/저장/댓글/알림·탈퇴(회원/의사묶음) 정상.

### Phase 1-B — H-1 신중 배치 (배포 순서 엄수)
**반드시 코드 먼저 → 그다음 REVOKE.** 한 마이그에 몰지 않음.
- **B단계-1 (코드+RPC 배포, GRANT 회수 없음)**:
  - `get_my_pii()` / `get_profile_private(target)` SECURITY DEFINER RPC 신설(본인=원본, 타인=field_visibility 필터).
  - own/타인 PII 읽기 전환: **`identity-server.ts`(birthdate 제거/RPC — 최우선)**, `middleware.ts`(안전 RPC + **fail-open 두 경로를 온보딩 redirect 로 fail-closed**), `onboarding/page`, `my/page`, `[handle]/page`(본인 settings + 타인), `admin/users`, `today/page`, `auth/callback`.
  - 배포 후 프로덕션 검증(로그인/온보딩/설정/마이/프로필/관리자 정상).
- **B단계-2 (검증 후 REVOKE)**: `REVOKE SELECT (birthdate, contact_email, gender, face_shape, skin_type, skin_concerns, interested_procedures, fitzpatrick) ON profiles FROM authenticated`(마이그). B단계-1 이 프로덕션에서 정상 확인된 뒤에만.
