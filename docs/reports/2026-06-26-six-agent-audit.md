## 2. 적용 완료 (이번에 직접 수정·배포한 항목)

### 2-1. 접근성 — 키보드 포커스 링 복원 (WCAG 2.4.7) · 가장 영향 큼
- **현재(수정 전)**: `globals.css` 가 `button/a/[role=button]` 의 `:focus` 와 `:focus-visible` **모두**에 `outline:none !important` 를 걸어, `app.module.css` 의 `.root :focus-visible { outline:3px }` 키보드 포커스 링까지 무력화했습니다. 주석은 "접근성은 :focus-visible 로 충분"이라 적었으나 같은 규칙이 그 :focus-visible 까지 끄는 자기모순이었습니다.
- **원인**: 마우스 클릭 잔상(파란 선) 제거를 의도했으나 셀렉터가 키보드 포커스까지 포함.
- **수정**: `:focus:not(:focus-visible)` 로 좁힘 — 마우스/터치 포커스만 outline 제거, 키보드 포커스 링은 복원.
- **효과**: 하단 탭·칩·글쓰기 버튼·카드 액션 등 거의 모든 인터랙티브 컨트롤에서 키보드 포커스 위치가 다시 보입니다. 보조기기 사용자·스토어 접근성 심사 결함 해소. 마우스 사용자 경험은 불변.

### 2-2. 런타임 안전 — 방어 코드 5종
같은 코드베이스가 이미 쓰는 방어 패턴에서 누락된 곳들을 정합화했습니다.
- **ScrollManager**: `savePos` 의 `sessionStorage.setItem` 을 try/catch 로 보호. 카카오톡/구글 인앱 브라우저 sandbox·QuotaExceeded 시 **네비게이션마다 화이트스크린/크래시**하던 위험 제거(SNS 유입 다수 경로).
- **admin/draft/publish**: 기존 카드 중복판정용 `row.meta` JSON.parse 를 try/catch 로 방어. 손상된 meta **1건이 발행 배치 전체를 500**으로 막던 단일 실패점 제거.
- **api/search/suggest**: GET 전체 try/catch + 빈 폴백(`{popular:[],cats:{}}`). DB 일시 장애 시 검색 발견 화면이 graceful degrade.
- **middleware**: `supabase.auth.getUser()` 를 try/catch 로 보호. 인증 엔드포인트 장애가 **전 사이트 라우팅을 막던** 시나리오 차단(하위 profiles 조회의 degrade 철학과 정합).
- **온보딩 만 14세 게이트**: 생년월일을 로컬 자정(`${birthdate}T00:00:00`)으로 파싱해 `today`(로컬)와 타임존 기준을 통일. 생일 당일 ±1일 **경계 오판** 제거(개인정보보호법 법정 연령 게이트 정확성).

### 2-3. 성능 — 피드 경로 네트워크 절감
- **피드 카드 me 판정 (N+1 제거)**: `ui.tsx` `useCardActions` 가 카드마다 `auth.getUser()`(=`/auth/v1/user` 네트워크 호출)를 발사하던 것을, 같은 파일의 `useCardViewer`·`PostCardMenu` 와 동일하게 **SSR SessionContext(`useSession`) 단일 출처**로 치환. 피드 초기 마운트 시 카드 수만큼 발생하던 호출과 첫 paint 후 me 깜빡임을 제거.
- **저장 토글 왕복 절감**: `useCardEngagement` 가 `toggle_card_save` RPC 직후 `cards.save_count` 를 **한 번 더 SELECT**하던 중복 제거. RPC 가 트리거 갱신 후 권위 카운트를 반환(마이그 0162/0030 확인)하므로 불필요. 저장 토글당 DB 왕복 1회 절감(좋아요 경로와 정합).

## 3. 권고 (제품 결정·대형 작업 — 디렉터 판단 필요)

자동 수정하지 않고 보고로 분리한 이유: 사용자에게 보이는 카피·기능 변경이거나, 프로덕션 DB·캐싱 구조를 바꾸는 큰 작업이라 디렉터 결정이 필요합니다.

### 3-1. [높음·성능] 댓글 미리보기 N+1 — 비정규화 컬럼 도입
- **현재**: 피드 카드가 뷰포트 근처에 오면 카드마다 `/api/comments?cardId=X&limit=50` 을 호출해 댓글 수 배지를 채웁니다(`cards` 에 `comment_count` 컬럼 없음). 무한스크롤로 카드가 쌓일수록 선형 증가.
- **권장**: `cards.comment_count` 비정규화 컬럼 + 트리거(like/save/share 와 동일 패턴) 추가 → 피드 select 에 포함해 배지를 fetch 없이 표시, 미리보기 댓글은 클릭 시 로드. **추가 마이그레이션 + 피드 RPC·댓글 표시 변경**이 얽혀 별도 검증이 필요합니다.
- **효과**: 피드 스크롤 중 댓글 API 호출이 카드 수만큼 → 0(수치)로 감소.

### 3-2. [중간·성능] 홈 ISR 전환 + 인기태그 캐시
- **현재**: 홈(`/`)이 `force-dynamic` 전면 무캐시라 매 방문 SSR(LCP ~2.1–2.6s). 또한 한 요청에서 `feed_cards_scored`(300행)를 피드용·인기태그용 **2회** 호출.
- **권장**: ADR 0020 "공유 셸 + 클라 개인화"를 홈에도 확장(짧은 revalidate ISR + 쿠키리스 anon 렌더, 좋아요/저장은 클라 분리). 인기태그는 `unstable_cache`(5~10분)로 분리해 두 번째 RPC 제거.
- **효과**: 홈 LCP/TTFB 가 캐시 HIT 시 상세 수준(0.4~1s대)으로 단축 + 서버리스 RPC·비용 절감.

### 3-3. [중간·UX] 온보딩 단계 분할·부분 저장
- **현재**: 회원가입 직후 강제 게이트가 7개 섹션·6개 필수값을 단일 스크롤 폼에서 한 번에 요구. 단계 표시·진행률·부분 저장 없음(오류는 맨 아래 한 줄, 빈 섹션 자동 스크롤 없음).
- **권장**: 필수 최소셋(생년월일·성별·동의)을 1단계로, 나머지는 "나중에" 옵션의 2~3단계로 분할 + 진행률 바 + 검증 실패 시 해당 섹션 스크롤·하이라이트.
- **효과**: 첫 진입 완주율 상승, 이탈 감소. (입력 대부분이 칩/탭 선택이라 심각도는 중간.)

### 3-4. [중간·정합] '쇼핑' 탭 죽은 탭 정리
- **현재**: 하단 5탭의 '쇼핑'이 라우팅 없이 토스트("준비 중")만 띄움. 별도로 `/shop` 준비중 페이지도 존재해 동작이 이원화.
- **권장**: 베타 동안 탭에서 제거(예: 검색/알림으로 대체)하거나, 최소한 `/shop` 으로 일관 라우팅 + "출시 알림 받기" 가치 행동 추가. **탭 구성은 제품 결정**이라 권고로 둡니다.

### 3-5. [중간·SNS표준] 알림 그룹핑 · 팔로우
- **알림 그룹핑**: 동일 (card_id, kind) 알림을 "○○님 외 N명"으로 묶는 집계 뷰/RPC. 인기 글에서 알림·푸시 도배 방지.
- **팔로우/구독**: 원장(doctor) 팔로우 최소버전 → 새 답변 알림 + /my 활동 피드. 인물 단위 리텐션 루프.
- 둘 다 **신규 기능**이라 제품 우선순위 결정 후 착수 권장.

### 3-6. [중간·유지보수] 네비게이션·신고사유·검색 진입 SSOT 통합
- **두 네비 공존**: 운영 `AppShell`(실제 노출) ↔ 레거시 `BottomNav`(거의 미노출)의 라벨·아이콘·쇼핑 동작 불일치. 탭 정의를 공용 상수로 단일화 권장.
- **신고 사유 4중 정의**: enum 값은 동일하나 한글 라벨이 화면마다 다름(harassment 3종 표기). `lib/report-reasons.ts` SSOT 로 통합 권장 — 단 **표준 라벨 카피 선택은 제품 결정**이라 보류했습니다.
- **검색 진입 3경로**: 헤더 인라인·풀스크린 `/search`·드롭다운으로 분산. 한 가지로 통일 권장.

### 3-7. [저위험·정리] 기타
- `/old-skin/*` 3개 라우트: "박제 백업"이라는 명시 보존 의도가 있어 임의 삭제 보류. 디자인 비교 용도가 끝났으면 제거 또는 `_archive` 이동 권장(제 판단으로 보존을 존중).
- 잔여 방어(useWeather name 계약 의존, recordView getUser, 카드 URL 인라인 8곳, escapeHtml 중복 등) — 저위험 정리 항목으로 차기 배치 권장.
- `npm audit` 취약점 17건(1 low·8 moderate·8 high) — 기존부터 존재. 별도 점검 권장(이번 leaflet 제거로 일부 표면 축소).
