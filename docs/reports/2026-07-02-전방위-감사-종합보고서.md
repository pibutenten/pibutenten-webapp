## 1. 🔴 CRITICAL — 즉시 조치 (디렉터 DB 실사로 확정)

### C-1. `current_active_profile_id()` 헤더 위조로 소유권 RLS 전면 우회 (IDOR / 수평 권한상승)

**두 독립 조사 에이전트가 각각 발견 → 디렉터가 라이브 DB로 확정.**

**무엇이 문제인가 (자연어)**
- 클라이언트(브라우저)는 Supabase REST(PostgREST)에 **앱을 거치지 않고 직접** 접속할 수 있습니다. anon 키는 `NEXT_PUBLIC_`으로 공개돼 있고, 로그인 회원은 본인 JWT를 갖습니다.
- 활성 명함(active identity)을 식별하려고 DB 함수 `current_active_profile_id()`가 HTTP 헤더 `x-active-profile-id`를 읽는데, **UUID 형식만 검사하고 그 명함이 호출자 본인 묶음인지 전혀 확인하지 않습니다.**

**디렉터 확정 근거 (프로덕션 DB 실물)**
```sql
-- current_active_profile_id() 본문 (실물)
v_active := v_headers ->> 'x-active-profile-id';
IF v_active !~ '^[0-9a-f]{8}-...$' THEN RETURN NULL; END IF;  -- 형식만 검사
RETURN v_active::uuid;   -- ← 묶음 소유권 검증 없이 그대로 반환
```
대조군인 `is_admin()`은 같은 상황에서 **`AND (p.id = uid OR p.auth_user_id = uid)`** 묶음 검증을 합니다. 즉 관리자 함수만 안전하고, **일반 회원 정책들은 무방비**입니다.

**실제로 뚫리는 정책 (프로덕션 `pg_policies` 실물 확인)** — 모두 `COALESCE(current_active_profile_id(), auth.uid())` 패턴:

| 테이블 | 정책 | 위험 |
|---|---|---|
| `diaries` (시술일기: 병원·의사·가격·본문 = 건강 PII) | select/insert/update_own — **`auth.uid()` 게이트조차 없음** | 타인 일기 **읽기·수정·삭제** |
| `notifications` | select/update_own | 타인 알림 열람·읽음처리 |
| `comments` | insert/update_self/delete_self | **타인 명의로 댓글 작성·수정·삭제(위장)** |
| `card_likes` / `card_saves` | insert/delete + card_saves select | 타인 명의 가짜 좋아요·저장 조회 |
| `push_subscriptions` | ALL | 타인 푸시 구독 조작 |
| `cards` (owner_update/delete/user_own_post) | 소유자 계열 | 타인 글 수정·삭제 |

피해자 UUID는 `profiles_public_select`가 `qual=true`라 `/rest/v1/profiles?select=id,handle`로 **즉시 열거** 가능합니다. 서버측 `resolveActiveIdentity`/`identity-switch`는 묶음을 검증하지만, **앱을 우회한 raw fetch 공격에는 이 서버 검증이 개입할 여지가 없습니다.** 마이그 0325(2026-07-02)가 이 함수를 다뤘으나 anon EXECUTE만 손대고 이 소유권 검증 누락은 미패치입니다.

**권장 수정 (한 곳만 고치면 전 정책 동시 방어)**
`current_active_profile_id()`에 묶음 검증을 넣어, 헤더 UUID가 `(profiles.id = auth.uid() OR auth_user_id = auth.uid())`일 때만 반환하고 아니면 `auth.uid()` 폴백(또는 NULL). `is_admin()`과 동일한 게이트. 병행해 `diaries` 정책에 `auth.uid() IS NOT NULL` 게이트 명시. 적용 후 "위조 헤더로 타 명함 데이터 접근 거부" 회귀 테스트 추가.
