## A. 즉시 반영 (Critical — 베타 기간 중 또는 6/1 공개 직전까지)

법적/PII 위험이 직접 발생하거나, 코드 수정 비용이 작으면서 효과 큰 항목.

### A1. `profiles_public_select` RLS 정책 즉시 점검 — 의료 PII 공개 위험
- **현상**: 마이그레이션 폴더 전수 검색 결과 `CREATE POLICY profiles_public_select` 정의가 단 한 줄도 없음. 주석들(0059·0107·0109)은 모두 "qual=true 그대로"라고만 언급. 즉 정책이 **Dashboard에서 수동 생성**되었고 git에 미반영. 만약 `USING (true)`면 anon이 PostgREST로 `birthdate / contact_email / gender / face_shape / skin_type / skin_concerns / interested_procedures / liked_procedures` 전부 조회 가능 → 개인정보보호법 민감정보(건강정보) 무단공개.
- **조치**:
  1. `select * from pg_policies where tablename='profiles'` 즉시 확인.
  2. anon은 컬럼 화이트리스트 view(`public_profiles_view`)만 접근, base table SELECT는 self + 같은 묶음 + admin으로 좁히는 마이그레이션 `0119_profiles_anon_column_lockdown.sql` 추가.
  3. `[handle]/page.tsx` 등의 select 컬럼 목록을 owner/anon 분기.

### A2. Push webhook secret 재로테이션 + 정책화
- **현상**: `0086_push_webhook_trigger.sql:25` 평문 secret 노출은 인지(0103에서 Vault 이관)되었으나, **0103 본문에도 새 secret(`73qpLp…`)이 평문 SQL 리터럴**로 박혀 git history에 영구 기록. `SECURITY.md`에는 0086만 언급. 동일 anti-pattern 반복.
- **조치**:
  1. `secrets.token_urlsafe(48)`로 신규 시크릿 발급 → Vault + Vercel env 갱신.
  2. 향후 vault 갱신 마이그레이션은 `current_setting('app.tmp_secret')` 패턴으로 본문에 값 미포함.
  3. `SECURITY.md`에 0103 노출 이력 추가.

### A3. 14세 미만 가입 차단 — DB 강제 추가
- **현상**: `signup/SignupForm.tsx`는 self-attestation 체크박스 1개, `onboarding/OnboardingClient.tsx:204-212`도 birthdate 유효성만 검사하고 만 14세 미만 차단 로직 없음.
- **조치**:
  1. onboarding 클라이언트에 `(now - birthdate) < 14년 → reject`.
  2. `ALTER TABLE profiles ADD CONSTRAINT chk_min_age CHECK (birthdate IS NULL OR birthdate <= now()::date - interval '14 years')`.

### A4. Admin KPI RPC에 `is_admin()` 가드 누락
- **현상**: `get_users_kpi / get_top_visitors / get_top_cards_by_views/_by_shares / get_admin_kpi / get_top_qas_by_*` 모두 `SECURITY DEFINER + GRANT EXECUTE TO authenticated`인데 본문에 `is_admin()` 검사 없음. 일반 로그인 사용자가 RPC 직접 호출 시 전체 회원 KPI·display_name+handle+visit_count가 노출됨.
- **조치**: 각 함수 도입부에 `IF NOT public.is_admin() THEN RAISE EXCEPTION USING ERRCODE='42501'; END IF;`. 마이그레이션 1개로 일괄 패치.

### A5. Naver OAuth — 동일 email 자동 계정 매칭 차단
- **현상**: `auth/naver/callback/route.ts:97-101`에서 `auth.users.email` 일치하면 자동으로 기존 user에 link. Google·Kakao로 가입한 사용자의 email을 Naver 측 미인증/변경 이메일이 가로채면 **계정 인수** 가능.
- **조치**: `auth.identities`에서 provider 비교. `provider='naver'`가 아니면 "이미 [다른채널]로 가입됨 → 기존 채널 로그인 후 [설정→계정 연결]에서 연결" 분기. 자동 매칭은 같은 provider일 때만.

### A6. `/api/preview-link` SSRF 우회 차단
- **현상**: `fetchWithTimeout`이 `redirect: "follow"`(line 67). 1차 hop만 `isHostSafeForExternalFetch` 검증, 이후 302 redirect로 사설망 IP로 우회 가능. og-extract는 `safeFetchExternal`로 hop별 재검증함. 또한 http도 허용, 응답 크기 cap 없음, Innertube 호출은 가드 미통과.
- **조치**: `safeFetchExternal`로 통일 또는 `redirect: "manual"` + hop별 재검증. https only, 응답 1~3MB cap.

### A7. 자기 자신의 admin 강등 차단
- **현상**: `/api/admin/users/[id]/role`이 마지막 admin이 자신을 user로 강등 가능 → admin 부재 상태.
- **조치**: `id === guard.userId && role !== 'admin'` 차단 또는 "마지막 admin" 검사. 한 줄짜리 패치.

### A8. Rate limit 인프라(0105)를 실제 적용
- **현상**: `check_and_increment_rate_limit` RPC가 `find_duplicate_profiles` 1곳에만 적용. 나머지 hot path 0건.
- **조치 (가벼움)**: `src/lib/rate-limit.ts` 헬퍼 하나 만들고, 우선 5개만:
  - `/api/og-extract`, `/api/preview-link` — user 30/분
  - `/api/comments` POST — user 10/분
  - `/api/articles` POST — user 5/분
  - `/api/auth/naver/start` — IP 10/분
  - `/api/upload` — user 20/분
- 그 외 admin draft 류는 admin 게이트로 충분.

### A9. CSRF Origin 허용 패턴 좁히기
- **현상**: `middleware.ts:82`의 `*.vercel.app` 와일드카드 → 임의 vercel 배포에서 cross-origin POST 가능. `:87` LAN IP 하드코딩.
- **조치**: `pibutenten-webapp-*.vercel.app`로 좁힘. LAN IP는 `NODE_ENV !== 'production'` 가드.

### A10. 사용자-facing error 메시지 일반화
- **현상**: `articles/route.ts:309`, `comments`, `og-extract/route.ts:104`, `me/delete/route.ts:75-82`, `auth/naver/callback/route.ts:196-200` 등이 `error.message`/`anonErr.message`를 클라이언트(또는 redirect URL)에 그대로 노출. Naver callback은 referer로 다음 페이지로도 유출.
- **조치**: 사용자 응답은 표준 문구, 상세는 `console.error`만. error_id(uuid) 발급 후 ID만 노출.

### A11. `pibutenten_onboarded` 쿠키 `secure: true` 추가
- **현상**: `middleware.ts:226` set 시 secure 미지정 → production HTTPS에서 자동 secure 처리 안 됨.
- **조치**: `secure: process.env.NODE_ENV === "production"` 추가. 한 줄.

### A12. `src/lib/supabase/admin.ts` 상단에 `import "server-only";`
- **현상**: 클라이언트 컴포넌트에서 실수로 import해도 빌드는 통과(런타임 호출 시점에야 throw). server-only import로 빌드 단계 차단.

## B. 장기적 반영 (의미 있지만 비용·범위 큼)

### B1. CSP enforce 전환
- 현재 Report-Only 사유로 "GoogleBot이 CSP로 차단된 리소스를 못 읽어 SEO 영향"이라 명시되어 있는데, **CSP는 브라우저 측 정책이며 검색엔진 크롤러는 CSP 적용 대상이 아님**. 사유 자체가 사실관계상 부정확할 가능성이 큼. 결정 재검토 후 단계적 enforce 권장.
- 단, `script-src 'unsafe-inline' 'unsafe-eval'` 제거(nonce 기반)는 Turbopack/JSON-LD 회귀 큼 → 우선 enforce만 시도, nonce는 별도 마일스톤.

### B2. Admin audit_log 도입
- 의료 도메인 분쟁/규제 대응 + 내부자 위협 방어. 테이블 1개 + 헬퍼 1개. role 변경, doctor 매핑, 카드 발행/삭제, 익명화 5개 라우트만 우선.
- 스키마: `audit_log(actor_id, action, target_table, target_id, before jsonb, after jsonb, ip, ua, created_at)`.

### B3. cards 본문 PII 검출 경고 UI
- WriteClient/admin EditClient 클라이언트단에 전화/이메일/주민번호 정규식 검출 → "개인정보가 포함되어 있어요" 토스트. 차단은 X, 경고만. 책임 분담 효과.

### B4. notifications INSERT deny 정책 명시
- 현재 정책 0개 + GRANT 0 = deny-by-default로 동작 중이지만, future 마이그레이션 사고 방지용으로 `FOR INSERT WITH CHECK (false) TO anon, authenticated` 명시. trigger의 SECURITY DEFINER가 정책 우회.

### B5. retention / 핫테이블 정리
- `card_views`, `card_impressions`, `card_shares`는 봇 트래픽으로 row 폭증 위험. 또한 `api_rate_limits`도 누적.
- 90일 또는 180일 보존 + `pg_cron` daily cleanup. (PIPA 접속기록은 별도 정책)
- 추가로 view/impression INSERT에 session+IP rate limit.

### B6. profiles `deleted_at IS NULL` RLS 명시
- 현재 응용 코드 책임. 누락 시 탈퇴자 마스킹 row 노출. RLS USING에 `deleted_at IS NULL OR is_admin()` 추가.

### B7. CSP `img-src https:` 와일드카드 축소 + `report-to` 설정
- Supabase Storage + youtube 도메인 외 제거. `Reporting-Endpoints` + `report-to`로 위반 로그 수집(Vercel KV 등).

### B8. RLS `WITH CHECK` 페어 일괄 점검
- 0099 등 다수 정책에서 USING만 있고 WITH CHECK 누락된 케이스 확인. row hijack 방어 차원.

### B9. preview-link 입력 cap + jsdom 입력 cap
- 응답 1~3MB로 통일. jsdom DoS 방지.

### B10. push 운영
- payload `message` 길이 cap(1000자), `VAPID_SUBJECT` env 필수화, 사용자별 push 구독 row cap(10개), endpoint hostname 화이트리스트(OPS #1).

### B11. `find_duplicate_profiles` rate limit 강화
- 60s/10회 → 60s/3회 + 일일/30회. 입력 email이 본인 auth email과 일치할 때만 매칭 결과 반환(타인의 (이메일·생일·성별)로 provider enumeration 차단).

### B12. zod 점진 도입
- API_POLICY.md 정책대로 신규 라우트부터. `comments`, `articles`, `admin/draft/publish` 우선. 입력 검증 코드 라인 수 30% 감소 효과도 기대.

### B13. gif 처리 일관성
- sharp `animated: true`로 재인코딩하거나 프레임/총크기 cap. 보안보단 일관성 이슈.

### B14. `articles` 버킷 IaC화
- Dashboard에서 수동 생성된 articles 버킷의 RLS/MIME/사이즈 한도를 마이그레이션으로 코드화.

### B15. Naver token exchange POST 전환
- GET + 쿼리스트링으로 client_secret이 access log/referer에 기록될 수 있음. Naver spec POST 지원 → 전환.

### B16. avatars 버킷 한도 5MB → 512KB

## D. 안 해도 될 것 (이미 충분히 잘 되어 있음)

코드 검토에서 모범 수준으로 확인된 영역이라, 더 손대지 않는 게 좋습니다.

### D1. 업로드 보안 (`/api/upload`)
- sharp 재인코딩 + EXIF 자동 제거 + 매직바이트 검증 + MIME 화이트리스트 + UUID 파일명 + 사용자 폴더 격리 — **헬스케어 모범 사례 수준**.

### D2. SSRF guard (`src/lib/ssrf-guard.ts`)
- IPv4/IPv6 사설 대역 + CGN + 클라우드 메타데이터 + DNS 재해석 + redirect hop별 재검증 + size/timeout cap + 프로토콜 화이트리스트. **이미 본격 구현**. preview-link만 이 가드를 안 쓰는 게 문제(A6).

### D3. service_role 키 격리
- `admin.ts` 단일 채널 + `NEXT_PUBLIC_` prefix 없음 + Route Handler에서만 사용. 직접 `createClient(URL, SERVICE_ROLE_KEY)` 호출 0건. `server-only` import 한 줄만 추가하면 완벽.

### D4. Webhook secret timing-safe 비교
- `push/send/route.ts:51-60`, `naver/callback/route.ts:11-20` 모두 길이 mismatch도 dummy compare로 timing 누설 방어. 모범 구현.

### D5. Prompt Injection 1차 방어
- `<` `>` 치환 + `<untrusted_input>` 격리 태그 + 정적 system prompt + `max_tokens` 명시. defense-in-depth 잘 갖춤.

### D6. 회원 탈퇴 흐름
- typed confirmation("탈퇴에 동의합니다") + in-place 익명화 + auth.users.delete + handle 마스킹 + PII 컬럼 일괄 NULL/{}. admin audit log(B2)만 추가하면 완성도 100%.

### D7. CSRF 다층 방어
- 미들웨어 Origin/Referer + SameSite=Lax + Naver state cookie + webhook secret + CSRF 면제 목록 명시.

### D8. OAuth Google/Kakao
- Supabase 기본 PKCE + `sanitizeNext()` open redirect 방어 다층 + `verifyOtp` 명시 호출.

### D9. 묶음(group-aware) RLS 패턴
- `same_group_profile_ids(auth.uid())` 헬퍼로 멀티 프로파일 통일 처리. 우수.

### D10. Vault 기반 secret 인프라
- 0103에서 push_webhook_secret을 vault로 이관. 패턴 자체는 모범 — secret 평문 SQL 잔존(A2)만 정리하면 됨.

### D11. RBAC 분리 (admin / doctor / user)
- `requireAdmin / requireAdminOrDoctor / requireActiveSuperAdmin / requireActiveSuperOrDoctorAdmin`로 묶음(bundle)·active identity 분리. 일반 SNS 표준 이상.

### D12. 기본 보안 헤더
- HSTS 2년 + preload, X-Frame-Options DENY, X-Content-Type-Options, Referrer-Policy, Permissions-Policy(기본 3개). 추가는 B7에서 점진 보완.
