## 0. 한눈 요약 (Executive Summary)

전반적으로 **AI 빌드 서비스 기준으로 매우 견고한 상태**입니다. 체크리스트가 "사고가 집중된다"고 경고한 **2번(RLS)·3번(시크릿)** 은 모두 통과했습니다. 이미 8차례 이상의 자체 보안 점검(2026-05-16 ~ 06)을 거친 흔적이 코드·마이그레이션·문서에 일관되게 남아 있습니다.

### 결론
- **P0 (오픈 전 반드시): 신규 발견 없음.** 체크리스트의 모든 P0 항목 통과.
- **단, 실제 취약점 1건 확정** — `award_points` RPC 가 누구나(비로그인 포함) 호출 가능하여 활동점수·레벨 조작 가능. 현재는 금전 가치가 없어 **P2**, **포인트몰·결제 오픈 시 P0 로 격상**.
- **결제·커머스(카테고리 7) 는 현재 미구현** (`/shop` 은 placeholder). 결제 코드 머지 직전 별도 리뷰 필수.

**조치 필요 (P2 → 결제 오픈 시 P0):**

> **`award_points(p_user_id, p_action, p_points, ...)` 임의 호출 가능**
> - **현상**: 함수 권한(proacl)이 `{=X, authenticated=X}` — **PUBLIC(anon 포함) + 로그인 사용자 모두 EXECUTE 가능**. 내부에 `auth.uid()`/`is_admin()` 가드 **전무**, 파라미터 `p_user_id` 를 그대로 신뢰.
> - **악용 시나리오**: 누구나 `supabase.rpc('award_points', { p_user_id: '임의_profile_id', p_action: 'x', p_points: 999999 })` 직접 호출 → 자신 또는 **타인**의 `activity_score`·`level` 조작(`activity_points` INSERT + `profiles` UPDATE).
> - **영향도**: 데이터 무결성 침해. 레벨/뱃지는 커뮤니티 신뢰 표시에 영향. **금전 가치·PII 유출은 없음**(무료 서비스)이라 현재 P2. 단 포인트가 적립·차감되는 커머스 오픈 시 즉시 **P0**.
> - **수정 영향 0**: 코드 전수 grep 결과 `award_points` 직접 호출처 없음 → 트리거·다른 RPC 내부 헬퍼로만 사용. SECURITY DEFINER 는 owner 권한으로 실행되므로 PUBLIC EXECUTE 를 회수해도 내부 호출은 그대로 작동.
> - **SECURITY.md §향후 점검 권고에 이미 인지된 항목**("임의 호출 가능 RPC 점검")의 잔여 미정리분. 0274/0276 에서 `recalc_user_level`·`find_other_auth_user_by_email` 만 정리됨.

### 5. 프론트엔드·XSS·보안 헤더 — 통과 (1건 개선)

- **XSS**: `dangerouslySetInnerHTML` 전 사용처가 **JSON-LD(`jsonLdString()` 으로 `<>&` 이스케이프)** 또는 **정적 SVG 상수**. 사용자 콘텐츠 직접 주입 0건. 사용자 본문은 평문 저장 + React 자동 이스케이프 렌더.
- **CSRF**: `middleware.ts` 가 unsafe method 에 Origin/Referer 화이트리스트 검증 + 쿠키 SameSite.
- **보안 헤더**: `next.config.ts headers()` 가 전 라우트(`/:path*`)에 적용 — `X-Frame-Options: DENY`·`X-Content-Type-Options: nosniff`·HSTS(preload)·`Referrer-Policy`·`Permissions-Policy`·COOP/CORP·`frame-ancestors 'none'`. (참고: `vercel.json` 엔 보안 헤더가 없으나 next.config 가 전역 적용하므로 **중복 불필요** — 누락 아님)

**개선 (P1):**

> **CSP 가 `Content-Security-Policy-Report-Only` 모드**
> - 위반을 **차단하지 않고 로그만 수집**. 게다가 `script-src` 에 `'unsafe-inline' 'unsafe-eval'` 포함 → enforce 로 바꿔도 인라인 스크립트 XSS 는 못 막음.
> - **의도된 보류**(주석상 SEO 우선 — enforce 시 GoogleBot 리소스 차단 우려).
> - **권장**: ① 위반 로그 충분히 수집 후 enforce 전환, ② JSON-LD 를 nonce 기반으로 전환해 `'unsafe-inline'` 제거. XSS 자체는 React 이스케이프 + 입력 검수로 1차 방어되므로 **긴급도는 중간**.

## 3. 수정 계획 (우선순위순)

### P1 — 빠른 시일 내

#### P1-1. CSP enforce 전환 검토
- **현황**: `next.config.ts` CSP 가 Report-Only(미차단) + `script-src 'unsafe-inline' 'unsafe-eval'`.
- **계획**:
  1. Report-Only 위반 로그(`/api/csp-report`) 1~2주 수집·정리하여 정상 리소스 화이트리스트 확정.
  2. JSON-LD 를 **nonce 기반**으로 전환 → `'unsafe-inline'` 제거.
  3. `Content-Security-Policy`(enforce)로 키 전환. GoogleBot SEO 영향 모니터링.
- **영향 범위**: `next.config.ts` 1파일 + JSON-LD 삽입 컴포넌트(nonce 주입). 빌드·렌더 영향 점검 필요.
- **회귀 주의**: GA4·Naver Analytics·YouTube 임베드 도메인 누락 시 기능 차단 → enforce 전 전수 확인.

#### P1-2. USER 전용 쓰기 API 온보딩 게이트 재검증 확대
- **현황**: USER role 의 온보딩(birthdate/terms) 재검증이 `/api/articles` POST 에만. middleware 가 1차 차단하나, 직접 API 호출 시 우회 가능성(이론적).
- **계획**: `/api/comments`·`/api/reviews`·`/api/diaries` 등 USER 작성 라우트에 동일 재검증(`getIdentityContext()` 후 role=USER 면 birthdate/termsAgreedAt 확인) 공통 헬퍼로 추가.
- **영향 범위**: 5~6개 라우트 + 공통 가드 헬퍼 1개. defense-in-depth(현 노출 위험 낮음).

#### P1-3. (결제 오픈 시) 카테고리 7 전체 별도 리뷰
- 결제 코드 머지 **직전** 게이트. 서버 금액 재계산·멱등키·PG 웹훅 서명·포인트 트랜잭션 원자성·카드정보 비저장·WebView 결제 스킴. 본 보고서 P2-1(award_points)도 이 시점 P0 로 격상.

### P2 — 운영하며 개선

#### P2-1. `award_points` PUBLIC EXECUTE 회수 (포인트 도입 전 필수)
- **계획**: 마이그레이션으로
  ```sql
  REVOKE EXECUTE ON FUNCTION public.award_points(uuid, text, numeric, text, text, integer) FROM PUBLIC, anon, authenticated;
  ```
  적용 후 `proacl` 에 `service_role`(+ owner)만 남는지 검증. 동일 패턴 함수 전수 재점검(`auth_uid_to_profile_id` 등 읽기 함수는 노출 데이터가 공개 매핑이라 위험 낮으나 함께 정리 권고).
- **영향 범위**: DB 마이그레이션 1개. 코드 직접 호출처 없어 **기능 영향 0**. `DATABASE.md`·`SECURITY.md` 동기화.
- **회귀 주의**: 트리거/내부 RPC 가 `award_points` 를 호출하는 경로는 definer 권한으로 유지되므로 정상 작동(적용 후 일일 로그인 적립·좋아요 적립 동작 확인).

#### P2-2. 로그아웃 시 identity 쿠키 명시 제거
- `signOut()` 직후 `pibutenten:identity`·`pibutenten:identity-mirror` 즉시 만료. 영향: 로그아웃 후 잔여 신분 상태 정리(부작용 거의 없음).

#### P2-3. CAPTCHA 도입 검토
- 가입·작성 단계 봇 방지. 공개(SITE_PUBLIC) 직후 봇 트래픽 관측 후 hCaptcha 무료 플랜 등 도입 판단.

#### P2-4. 민감정보(건강) 단계별 동의 강화
- 시술후기 작성 폼에 건강정보 수집·공개 범위 동의 체크 추가, 처리방침과 정합.

#### P2-5. 옛 API 라우트 zod 마이그레이션 완료
- 수동 검증 라우트(og-extract·preview-link·push/subscribe 등)를 zod 스키마로 통일.

#### P2-6. npm audit·에러추적 정기화
- `npm audit` CI 편입 + Dependabot. Sentry/업타임 모니터링 도입 검토.
